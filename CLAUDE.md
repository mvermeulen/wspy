# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

wspy ("workload spy") is a C instrumentation wrapper: it launches a child workload as a subprocess and
collects runtime + hardware performance counter metrics around it (IPC, topdown, cache, TLB, branch,
software counters), plus optional system-wide (`/proc`, `/sys`) and AMD/NVIDIA GPU metrics. Output is
CSV or human-readable. This is the author's internal research testbed, published for portability across
machines â€” treat it as a small, pragmatic C codebase rather than a polished library.

## Build & Test

- `make` â€” builds `wspy`, `cpu_info`, `proctree`, `wspy-validate`, `wspy-ledger`, `wspy-store`, `wspy-summary`, `wspy-plot` (default, no GPU support). `wspy-store`/`wspy-summary` link against the system SQLite library (`-lsqlite3`) â€” needs `libsqlite3-dev` (or your distro's equivalent) installed to build; `wspy`/the other tools (including `wspy-plot`) have no such requirement. `wspy-plot` shells out to a `gnuplot` binary at runtime (not a build-time dependency) â€” see its entry below.
- `make AMDGPU=1 [ROCM_DIR=/path/to/rocm]` â€” builds with AMD GPU support, also builds `amd_smi`, `amd_sysfs`. If `ROCM_DIR` isn't given, the Makefile auto-detects it by checking for `amd_smi/amdsmi.h` under `/opt/rocm` (the traditional ROCm install location) then `/usr` (where distro packages like Debian/Ubuntu's `rocm-smi-lib` install it), preferring `/opt/rocm` if both exist
- `make NVIDIA=1` â€” builds with NVIDIA GPU support (`--gpu-nvidia`/`--gpu-nvidia-device`), also builds `nvidia_nvml`. Unlike `AMDGPU=1`, this has no build-time header/library dependency to auto-detect â€” `nvidia_nvml.c` `dlopen()`s `libnvidia-ml.so.1` (the NVIDIA driver's runtime library) at *run* time, with a small hand-vendored subset of NVML's stable C ABI declared locally instead of requiring `nvml.h`/the CUDA toolkit at build time. `AMDGPU` and `NVIDIA` are independent axes â€” `make AMDGPU=1 NVIDIA=1` builds both (useful on a host with both an AMD iGPU and an NVIDIA dGPU, e.g. a laptop APU + discrete GPU)
- `make test` â€” builds and runs `test_wspy`, `test_proctree`, `test_validate`, `test_ledger`, `test_ibs`, `test_power`, `test_phase`, `test_store`, `test_summary`, `test_plot`, and `test_affinity` (unit-style tests; `test_wspy.c` `#include`s `wspy.c` directly with `TEST_WSPY`/`AMDGPU=0`/`NVIDIA=0` defined to stub out `main()` and disable GPU code â€” the `test_wspy` Makefile target always compiles its own objects with `-DAMDGPU=0 -DNVIDIA=0`, independent of whatever the top-level `AMDGPU`/`NVIDIA` values are, so it can't pick up a mismatched GPU-enabled `topdown.o`/`wspy.o` from the same build tree; `test_validate.c` and `test_ledger.c` follow the same `#include "validate.c"`/`#include "ledger.c"` + `-DTEST_VALIDATE`/`-DTEST_LEDGER` pattern to reach each tool's static check functions; `test_ibs.c` similarly `#include`s `ibs.c` directly (no `TEST_IBS` macro needed since `ibs.c` has no `main()` to stub) and builds a fake `/tmp/test_ibs_sysfs/{ibs_fetch,ibs_op}/...` directory tree to exercise `ibs_probe_at()`'s sysfs-scanning logic without depending on real IBS hardware; `test_power.c` follows the same shape (`#include`s `power.c` directly, no `TEST_POWER` macro needed) against a fake `/tmp/test_power_sysfs/{power,power_core}/...` tree, exercising `power_probe_at()`'s sysfs parsing (event/scale/unit files, format-field bit location) and `power_build_pkg_event()`'s bit-placement math without depending on real RAPL-capable hardware, plus a structural (host-dependent-outcome-tolerant) sanity check of `power_counter_group()` itself, which always probes the real sysfs path; `test_phase.c` follows the same direct-`#include "phase.c"` pattern, stubbing the handful of `wspy.h` globals (`interval`, `phase_flag`, `counter_mask`, `aflag`, `outfile`) and topdown.c's `find_ci_label()` it needs rather than linking the whole program, and drives `phase_detector_update()`'s state machine directly with synthetic IPC sequences to exercise warmup/steady/degraded transitions, hysteresis, and baseline drift without needing real perf counters; `test_store.c` follows the same direct-`#include "store.c"` + `-DTEST_STORE` pattern, using an in-memory (`:memory:`) SQLite database to exercise schema idempotency, upsert/re-ingest without duplication, `(hostname,run_id)` collision detection, manifest-enrichment matching/mismatch/missing-path handling, incremental byte-offset tracking on a growing run-index file, CSV metric-value ingestion (aggregate/`--interval`/`--per-core` shapes, percent and `nan`/`-nan` cell handling, per-row column-count-mismatch skipping, re-ingest idempotency, skip conditions) against synthetic CSV fixtures, and a hand-built v1-shaped database migrating cleanly to the current schema version via `ensure_schema()`, all without needing real `wspy` runs; `test_summary.c` follows the same direct-`#include "summary.c"` + `-DTEST_SUMMARY` pattern, using a minimal hand-built `:memory:` `runs`/`metric_values` fixture (not `store.c`'s full `SCHEMA_DDL`, to stay decoupled from that schema's own evolution) to exercise `compute_stats()`'s min/max/mean/median/stddev/outlier-flag math directly, `summarize()`'s per-run `AVG()` collapse and (group,metric)-bucket streaming against command/hostname/metric filters, `--min-runs` thin-bucket skipping, `cpu_vendor`-grouping's `NULL` â†’ `"(unknown)"` bucket, `NULL`-only-metric exclusion, CSV field quoting, `--show-runs`' contributing-run-list column (present/absent, plus its `(+N more)` truncation marker under a large enough bucket), `trace_run()`'s artifact-path resolution against a real temp directory (existing files/dirs, stale/never-existed paths, a bare relative `output_path`'s plots-dir degradation, an unmatched `(hostname,run_id)`), and `open_summary_db()`'s schema-version gate against a real on-disk file (needed there specifically since it opens read-only, which a fresh `:memory:` connection can't meaningfully exercise); `test_plot.c` follows the same direct-`#include "plot.c"` + `-DTEST_PLOT` pattern, driving `match_templates()`/`add_fallback_match()`/`process_csv()` against synthetic CSV header strings and small on-disk CSV fixtures (amdtopdown-shaped, systemtime-shaped, an aggregate/no-`time`-column shape, an all-dimension shape, extra unclaimed columns) to exercise template matching, the generic fallback, and the "no time column"/"empty file" skip conditions -- entirely without gnuplot installed, since actually invoking gnuplot is only exercised by hand/in `run_tests.sh`'s environment, not this unit suite; `test_affinity.c` follows the same direct-`#include "affinity.c"` pattern (no `TEST_AFFINITY` macro needed, `affinity.c` has no `main()` to stub), building a fake `/tmp/test_affinity_sysfs/cpu<N>/{topology,cache/index3}/...` directory tree to exercise `affinity_topology_discover_at()`'s SMT-sibling/L3-domain parsing (mirroring `test_ibs.c`'s `ibs_probe_at()` fixture approach), plus `affinity_parse_spec()`/`affinity_format_cpu_set()`/`affinity_resolve()` (including partial- and zero-overlap against a hand-built `cpu_info`'s `is_available` mask) without needing real hardware topology or `sched_setaffinity()` privileges)
- `./run_tests.sh` â€” full test script: builds, runs unit tests, then integration-smoke-tests the real `wspy`/`proctree` binaries (CSV output, `--tree` output, network interface naming, GPU flag warnings when built without `AMDGPU=1`/`NVIDIA=1`, `wspy-run` profile launching), runs `tests/golden_output.sh`, `tests/capability_matrix.sh`, `tests/wspy_queue_smoke.sh` (job/queue lifecycle tests against fake wspy/wspy-run/wspy-plot/wspy-store binaries â€” build/GPU-axis-independent, so it runs once, not per-build), and `tests/arm_topdown_microbench.sh` (the ARM microbench script self-skips on non-ARM hosts or without perf access), and additionally exercises the AMDGPU build if ROCm headers are found under `/opt/rocm` or `/usr`, and the NVIDIA build if `libnvidia-ml.so` is resolvable via `ldconfig` (re-running the golden-output/capability-matrix `tests/*.sh` scripts against each build too â€” the two GPU axes are tested independently, not combined, since `run_tests.sh` always rebuilds the plain default in between)
- `tests/golden_output.sh` â€” golden output-contract tests (`INVESTIGATION.md`'s "Testing and documentation" track): pins the current, intentional shape of `wspy`'s output without needing root/perf access â€” exact-match CSV headers for the base/rusage columns and, vendor-gated via `./cpu_info`, every AMD raw-event counter group plus a small native multi-pass counter execution (`--passes=ipc,cache2`) bundle pinning the merged-row column order; a regex check for `--system`'s host-specific network-interface columns; a vendor-neutral generic check (CSV header/value column counts must match) run across the full flag matrix; a handful of always-present normal-output summary-line checks; and a `--tree` line-grammar/`ptrace-summary`-footer consistency check at a small, readable scale (`run_tests.sh` keeps its own larger 2000-process version as a stress/integrity test, not a format-contract one). Can be run standalone once `wspy`/`cpu_info` are built.
- `tests/capability_matrix.sh` â€” capability-matrix smoke tests (same `INVESTIGATION.md` track): runs ~35 flag-bundle combinations (one per counter group, GPU flags, per-core/interval/tree modifiers, manifest/run-index/`--capabilities`, a kitchen-sink combo, native multi-pass counter execution (`--passes`), and `--exit-with-child`'s two expected exit codes) and asserts each exits as expected, never prints `fatal error`, and shows no crash indication â€” the graceful-degradation contract. `--passes`' own incompatibility checks (`--interval`/`--per-core`/`--tree`/`--ibs-basic` combined with `--passes`) are the one deliberate exception, asserted via a separate `run_expected_fatal_bundle()` helper that requires a fatal error rather than forbidding one, since these bundles are supposed to be rejected outright. Auto-detects CPU vendor (`./cpu_info`) and both GPU builds (whether `--gpu-busy`/`--gpu-nvidia` warn "GPU support not built") and reports all three, since none of these axes is switchable from a single host/build; `run_tests.sh` calls it once per build it produces so every GPU-build axis gets covered. Known gap (documented in-script, not fixed): `--per-core` combined with any counter group produces a CSV header with only the base/coverage columns while each per-core row still appends that group's values, so the `per-core-topdown` bundle deliberately doesn't assert CSV column-count matching â€” see the comment there.
- `./test_amd_smi.sh` â€” focused smoke test for the `amd_smi` module (requires `AMDGPU=1` build)
- `./test_nvidia_nvml.sh` â€” focused smoke test for the `nvidia_nvml` module (requires `NVIDIA=1` build and a real NVIDIA GPU/driver â€” `nvidia_nvml.c` dlopen()s the driver's runtime library rather than linking at build time, so this is the one place that dependency is actually exercised)
- `make clean` â€” remove `.o`/backup files; `make clobber` â€” also remove built binaries
- No root required for `./cpu_info` (CPU detection only). Perf counters and `--tree` (ptrace) require root, or `CAP_SYS_PTRACE` + `perf_event_paranoid <= 1` â€” use `scripts/setup_perf.sh` to check/adjust `nmi_watchdog` and `perf_event_paranoid` sysctls. `--power`'s RAPL/`energy-pkg` access needs root or `CAP_PERFMON` specifically (stricter than `perf_event_paranoid` alone) â€” `scripts/setup_perf.sh` also checks/grants that on the `wspy` binary itself (`getcap`/`setcap`, `cap_perfmon+ep`), same check-then-prompt/`-y` convention as its sysctl checks; the grant is a file capability tied to that exact binary's inode, so it's dropped by every `make` rebuild and needs re-running.

## Development workflow

Starting with the 4.0 development cycle (see `INVESTIGATION.md`), new feature work happens on a
short-lived feature branch rather than committing directly to `master`. Small housekeeping changes
(typo fixes, doc tweaks, README updates) can still go straight to `master` â€” this applies to actual
feature/behavior changes, especially anything tracked in `INVESTIGATION.md`'s phase-tagged
priority lists.

- **Branch naming:** `feature/<slug>`, e.g. `feature/run-manifest`, `feature/gpu-path-scan`. Prefer
  one branch per inventory row/idea in `INVESTIGATION.md` rather than one branch per phase, so
  each PR stays reviewable and revertable independently.
- **Starting a feature:** `git checkout master && git pull && git checkout -b feature/<slug>`.
- **While working:** commit normally on the branch; rebase or merge `master` into the branch as
  needed to stay current, but don't rewrite history that's already been pushed.
- **Finishing a feature:** push the branch (`git push -u origin feature/<slug>`) and open a PR with
  `gh pr create` (origin is already `github.com/mvermeulen/wspy`). Merge through GitHub once it's
  ready â€” don't merge feature branches into `master` locally and push `master` directly.
- **Before opening the PR:** run `./run_tests.sh` (and `./test_amd_smi.sh`/`./test_nvidia_nvml.sh` if
  the change touches GPU code) â€” the branch should be in a state that passes the existing test suite. `run_tests.sh` now
  includes `tests/golden_output.sh`'s CSV column-order/summary-fragment/tree-format checks and
  `tests/capability_matrix.sh`'s flag-bundle graceful-degradation checks (see "Build & Test" above),
  so most "CSV vs. human output" contract regressions (see below) are caught automatically; extend
  those two files' tables when a change adds/reorders/removes a column rather than relying on manual
  checking alone.
- **Scope:** keep a feature branch to one inventory idea where practical; large phase-sized efforts
  (e.g. all of "Run artifact foundation") should still land as a series of small merged PRs rather
  than one long-lived branch, to avoid the drift and merge pain of a big-bang branch.

## Architecture

**Data flow:** CLI (`wspy.c`) â†’ CPU detection (`cpu_info.c`) â†’ counter setup (`topdown.c`) â†’ fork/exec
child with optional ptrace â†’ periodic/final counter reads â†’ CSV or human-readable output.

**Files and responsibilities:**
- `wspy.c` â€” `main()`, `parse_options()` (long/short option parsing into globals like `counter_mask`, `system_mask`, `aflag`/`sflag`/`csvflag`/`treeflag`), orchestrates setup â†’ start â†’ read â†’ print lifecycle. `--passes=<list>` (native multi-pass counter execution, `multipass.h`/`multipass.c` â€” `INVESTIGATION.md`'s "What shipped in 4.1") branches `main()` into `run_multipass()` instead: re-launches the workload once per automatically-sized pass from `multipass_plan_build()`, merging the result into one CSV/manifest/run-index record rather than requiring N separate `wspy` invocations the way `wspy-run`'s builtin profiles do today (see `wspy-run`'s own entry below â€” it isn't changed by this, `--passes` is a lower-level capability it could adopt later). Each pass gets its own `setup_counter_groups()`/`setup_counters()`/`start_counters()`/`launch_child()`/`wait4()`/`read_counters()` cycle exactly like the single-pass path, but MUST `close_counters()` (`topdown.c`) its fds before the next pass's `setup_counters()` opens new ones â€” nothing in this codebase closed a counter fd before this feature existed, since a single-pass run just exits with its fds open. `print_metrics()` needs no changes: every counter group is self-normalizing (each raw/cache group carries its own denominator counters, per `find_ci_label()`'s group-scoped lookups) and dispatches purely off `cgroup->mask`, so calling it once per pass into the same row just concatenates that pass's columns, with no possible name collision since the bin-packer never duplicates a `COUNTER_*` bit across passes. Because each pass is a genuinely separate re-execution of the workload (its own start/finish time, rusage, exit status), there's no single canonical value for those across N runs -- pass 0 ("primary") supplies the merged row's base columns and the manifest's top-level `exit_status`, while every pass's own timing/exit status/coverage delta is still recorded in the manifest's `passes[]` array for full audit (a differing exit status across passes warns, doesn't fail the run). V1 scope is aggregate-only: `--interval`, `--per-core`, `--tree`, `--ibs-basic`/`--ibs-memory-deep`, and the GPU flags are all fatal'd against `--passes` (checked right after `parse_options()` returns), since each either has no defined multi-pass merge semantics or already has its own separate collection/budget model. `populate_manifest_common()`/`exit_code_epilogue()` factor the manifest-population/`--exit-with-child` epilogue logic shared between `main()`'s single-pass tail and `run_multipass()`'s tail, so both stay in sync. `--multiplex` (opt-in, off by default) swaps `run_multipass()`'s plan builder from `multipass_plan_build()` to `multipass_plan_build_multiplexed()`: instead of bin-packing the requested groups into N passes that each fit the general-purpose PMU budget, it opens every requested group in one pass and lets the kernel multiplex whatever doesn't fit, trading precision (heavier multiplexing means lower per-counter `confidence_ratio()`) for a single workload execution â€” safe to offer now that `read_counters()`'s multiplex scaling (see its entry below) keeps an oversubscribed run's *values* correct, not just flagged as low-confidence. `--multiplex` given without `--passes` is fatal, since it has no meaning on its own. `--preset-name <name>`/`--config-name <name>`/`--config-option <k>=<v>` (repeatable) are purely metadata (`INVESTIGATION.md`'s "What shipped in 4.1", "structured configuration provenance") populating `manifest_info.config_provenance` (`manifest.h`) â€” `wspy` itself never reads them back or lets them affect the run, they only exist so a front end that *does* have a preset/configuration/option vocabulary (`wspy-run`, the web launcher) can record its own choice of words into the manifest/run-index rather than a report having to re-derive "what this was" from `counter_mask`/argv. A malformed `--config-option` (no `=`) is warned about and dropped, not fatal. `--affinity=<spec>` (`affinity.c`/`affinity.h`, `INVESTIGATION.md`'s "Core/thread affinity control" item, shipped ahead of its originally-planned phase) parses `all`/`thread=<id>`/`nosmt`/`domain=<id>`/`coretype=<id>`/`cpuset=<c0,c1,...>` into a `struct affinity_spec`; `main()` resolves it (`affinity_topology_discover()`+`affinity_resolve()`) right after `inventory_cpu()`/`check_nmi_watchdog()`, before either the `--passes` branch or the single-pass path, so both share one resolved `requested_affinity`/`affinity_active` pair that `topdown.c`'s `launch_child()` reads directly â€” an unresolvable request (out-of-range `thread=`/`domain=` id, or a `cpuset=`/`domain=` that shares no CPU at all with this process's own current affinity mask) is `fatal()` here, since a real run can't proceed without a real placement; a *partial* mismatch against that mask instead narrows to the overlap with a warning, same degrade-don't-fail idiom as `coverage.c`. `--list-affinity` (return code 5 from `parse_options()`, alongside `--capabilities`'/`--preflight`'s 3/4) is the standalone, no-privileges-needed discovery probe, and is also folded directly into `run_capabilities_probe()`'s own report. `populate_manifest_common()` records the resolved mode/requested-text/cpu-list into `manifest_info.affinity_mode`/`affinity_requested`/`affinity_cpus` unconditionally (even for the "all" default), so a report never has to cross-reference `cpu_info.num_cores` to know what "all" meant on that host. `--help`/`-h` prints `print_full_usage()` (a sectioned reference -- discovery probes, run control, process tree, run-artifact metadata, counter groups, IBS, power, GPU -- covering everything below) and exits 0; a real usage error (bad flag, missing command) instead prints a short two-line pointer to `--help` rather than dumping that whole reference, matching this codebase's general preference for a quick signal over a wall of text on failure. `--counters=<list>` (comma-separated names, reusing `multipass.c`'s `multipass_lookup_group_name()`/`multipass_group_names[]` table -- the same vocabulary `--passes=<list>` already established) is the recommended way to select counter groups now, ORing the named bits into `counter_mask` exactly like every individual flag already does (additive, not a replacement -- `--counters=topdown` still leaves the default-on `COUNTER_IPC` set unless `--no-ipc` is also given). Every one of the ~20 individual per-counter-group boolean flags this replaces (`--dcache`, `--topdown`, `--branch`, `--ipc`/`-i`, etc. -- not their `--no-<flag>` negations, which remain the only way to turn off a default-on group) now calls `warn_deprecated_counter_flag()` when used, a soft deprecation: the flag still works identically, it just now warns, so nothing breaks and a grep for the warning text finds every remaining old-style call site (`wspy-run`'s builtin profiles, `web/joblib.py`'s argv builders, and this repo's own `tests/*.sh` bundles all still use the old flags today and will warn until migrated). `--no-deprecation-warnings` (or the `WSPY_NO_DEPRECATION_WARNINGS` environment variable, any non-empty non-`"0"` value) silences these; `deprecation_warnings_should_suppress()` decides this via a small pre-scan of `argv` at the very top of `parse_options()`, before the `getopt_long()` loop that would otherwise fire a warning for an earlier deprecated flag before ever reaching `--no-deprecation-warnings` later on the same command line -- a per-flag "have I already warned" check can't fix this, since the problem is ordering, not repetition. Separately (an unrelated correctness fix found while touching this same code): `--no-phase-detect` and `--tree-connect` had been assigned `getopt_long()` `val`s (63, 83) that collide with `'?'` (the sentinel `getopt_long()` itself returns for any unrecognized option or missing required argument) and `'S'` (a stray, undocumented short option that was in the optstring with no case to handle it) -- either collision meant a genuinely bad/malformed flag matched the wrong `case` in `parse_options()`'s switch instead of falling through to `default: return 1` (the usage error), confirmed live: an unrecognized flag given alongside a real workload command printed `getopt_long`'s own "unrecognized option" line, then ran the workload anyway. Renumbered to unused values and dropped the dead `'S'` short option; `--dcache`'s negation, separately found to have been registered as `--no-cache` instead of `--no-dcache`, was also renamed to match (nothing depended on the old spelling).
- `topdown.c` (largest file, ~68K) â€” vendor event tables (`intel_raw_events[]`, `amd_raw_events[]`, `arm_raw_events[]`), `perf_event_open` wrapper, counter group constructors, `setup_counters`/`start_counters`/`read_counters`, all `print_*` formatters, `launch_child`/`ptrace_setup`/`ptrace_loop` (child launch + fork/exec/exit tracing), `check_nmi_watchdog`. `launch_child()`'s forked child applies core/thread affinity control (`affinity.c`/`affinity.h`) via `sched_setaffinity()` right after `fork()` -- before `PTRACE_TRACEME`/`execve` so it's inherited across the exec -- when `affinity_active` (set by `wspy.c:main()` after resolving `--affinity=<spec>` against the discovered topology) is nonzero; skipped entirely for the default `all` request (a documented no-op), and a failure here is logged (`error()`, which doesn't exit) rather than aborting the run, since the workload can still launch unpinned. ARM now has a topdown-equivalent decomposition wired through raw PMU events (`op_retired`/`op_spec`/`stall_slot*`/`stall_backend_mem`/branch events) rather than perf's metricgroup interface, and `setup_counters()` honors per-core `target_cpu` binding in `--per-core` mode so mixed ARM PMU clusters can route RAW events to each core's own PMU type. `ptrace_loop()`'s syscall-stop handling reads the traced process's registers through `ptrace_arch.h`'s `wspy_regs_t`/`ptrace_getregs()`/`PTRACE_SYSCALL_NUM`/`PTRACE_SYSCALL_ARG2` rather than `struct user_regs_struct`/`PTRACE_GETREGS`/`.orig_rax`/`.rsi` directly, so it isn't x86_64-hardcoded (see `ptrace_arch.h` below). `read_counters()` scales every counter's raw delta by that read's own multiplex ratio (`time_enabled`-delta/`time_running`-delta since the previous read, tracked via `struct counter_info`'s `last_time_running`/`last_time_enabled`, `cpu_info.h`) before storing it in `.value` (`INVESTIGATION.md`'s "What shipped in 4.1", the multiplex-scaling correctness fix) â€” a counter multiplexed off the PMU for part of a window previously reported its raw (undercounted) count with only the separate confidence envelope hinting at the loss; now the value itself is extrapolated to the full window, and `time_running`/`time_enabled` on `struct counter_info` hold that read's *delta* (not cumulative-since-start) for `confidence_ratio()`/`phase_current_ipc()` to gate on. `print_ipc()` and `phase.c`'s `phase_current_ipc()` used to redo this scaling themselves from raw `cpu-cycles`/`instructions` values; both were simplified to read the now-pre-scaled `.value` directly, since rescaling there would double-count it. `ptrace_loop()`'s `struct ptrace_pid_entry` (one per traced pid, in a small hash table) tracks each tracee's own syscall entry/exit state independently, replacing an earlier loop-global pair of variables that could misclassify entry vs. exit under concurrent syscall activity from more than one tracee. `--tree-futex` (`tree_futex`) uses it to time blocking `futex(2)` waits blocking read-side (`read`/`pread64`/`readv`/`preadv`/`recvfrom`/`recvmsg`) and write-side (`write`/`pwrite64`/`writev`/`pwritev`/`sendto`/`sendmsg`) syscalls passive `/proc/<pid>/schedstat` read at the same exit point, parsed into a `schedstat` line (`cpu_seconds`/`run_delay_seconds`/`nr_timeslices`, the middle field converted from the file's raw nanoseconds) and likewise emitted whenever the read succeeds rather than gated on nonzero. It closes a gap the "Critical-path/synchronization-latency deep-dive" (`INVESTIGATION.md`) called out: a degraded phase with no futex/io-wait activity was previously assumed to be a genuine hardware stall, but a runnable-but-not-scheduled process (CPU oversubscription, cgroup throttling) looks identical from counters alone â€” `run_delay_seconds` is what tells those two apart. `main()` calls `check_schedstat_enabled()` once, only when `tree_schedstat` is set (not on every run, unlike `check_nmi_watchdog()`): real-hardware testing found that trusting the `/proc/sys/kernel/sched_schedstats` sysctl directly is unreliable â€” something can force scheduler-stat collection on outside that sysctl's own write path, leaving the file reading `0` while `/proc/<pid>/schedstat` already returns real nonzero data â€” so the check instead reads this process's own `/proc/self/schedstat` and warns only if its `nr_timeslices` is genuinely zero (this process has certainly been scheduled at least once by the time `main()` gets here, so a real zero there means collection isn't happening, regardless of what the sysctl claims). Validated against a real CPU-oversubscription test (32 busy-spin processes on a 16-core host): `run_delay` totaled ~32.9s versus ~0.003s for the same test with only 8 (undersubscribed), confirming the signal tracks real contention. `--tree-vmsize` (`tree_vmsize`) is the same passive-read shape again â€” a fourth exit-time scrape, `/proc/<pid>/status`, parsed for `VmHWM` (peak RSS)/`RssAnon`/`RssFile`/`RssShmem`/`VmSwap` into a `vmsize` line. This flag existed in the CLI for a long stretch of this codebase's history as a pure no-op (registered, and even given a `parse_options()` case to avoid crashing, but `tree_vmsize` itself was never consumed by `topdown.c`/`proctree.c`) â€” rather than leave that dead flag in place and add a confusingly-named sibling, this is the flag actually being wired up for the first time. Combined with `--tree-io`'s existing `rchar`/`read_bytes` split, it separates three previously-indistinguishable "why is this process big/slow" stories: page-cache-bound (high `RssFile`/`rchar`, low `read_bytes`), genuinely allocation-heavy (high `RssAnon`/`VmHWM`), or swap-pressured (nonzero `VmSwap`). **Caveat confirmed via real testing:** `VmHWM` is a true kernel-tracked historical peak and stays accurate regardless of what a process does right before exiting, but `RssAnon`/`RssFile`/`RssShmem`/`VmSwap` are an *exit-time snapshot*, not the composition at peak -- a process that frees/unmaps memory during its own normal shutdown (e.g. CPython's interpreter finalization) will under-report those relative to its true peak composition even though `VmHWM` itself is unaffected (confirmed with two versions of a 200MB-allocation Python test program, one exiting via normal shutdown, one via `os._exit()`). `--tree-connect`/`--tree-wait`/`--tree-poll`/`--tree-nanosleep` (`tree_connect`/`tree_wait`/`tree_poll`/`tree_nanosleep`) ship the remaining four of the Critical-path deep-dive's six named syscall candidates in one PR (futex and blocking I/O were the first two) â€” mechanically each is just another `if (tree_X && (pid_entry->current_syscall == SYS_...))` branch in the same exit-stop dispatch `--tree-futex`/`--tree-io-wait` already use, accumulating into new `{connect,nanosleep,wait,poll}_{count,seconds}` fields on `struct ptrace_pid_entry` and flushed as `connect`/`nanosleep`/`wait`/`poll` lines (`<count> <seconds>`, only when nonzero â€” same idiom as `futex`) before the `exit` line. `connect` watches `SYS_connect` only (no sockaddr decoding â€” deferred, needs a bounded-length `ptrace_read_bytes()` generalization of the existing pathname reader); Default output is a human-readable report (added/removed subtrees, then changed nodes sorted by `|delta utime+stime|` descending, capped with a `(+N more)` marker mirroring `summary.c`'s own truncation convention); `--json` combined with `--diff` emits the merged diff tree instead, for `web/server.py`'s viewer. Exits 1 if any node differs, 0 if the two trees matched exactly (`diff(1)`'s own exit-code convention). `getopt_long()` (not plain `getopt()`) is used for `--json`/`--diff`/`--diff-threshold`, with enum values â‰¥256 for their `val`s â€” deliberately avoiding the exact `getopt_long` `val`-collision bug this file documents was found and fixed in `wspy.c` for `--no-phase-detect`/`--tree-connect`.
- `error.c`/`error.h` â€” centralized logging: `fatal()`, `error()`, `warning()`, `notice()`, `debug()`, `debug2()`, gated by `set_error_level()`
- `manifest.c`/`manifest.h` â€” writes the optional JSON run manifest (`--manifest <file>`): command line, start/finish timestamps, child exit status (when known â€” not available in `--tree` mode since `ptrace_loop` reaps children itself), host/CPU info, the option flags used, and the list of output files produced. `MANIFEST_SCHEMA_VERSION` in `manifest.h` is the SemVer of the manifest's JSON *shape*, independent of `WSPY_VERSION_MAJOR`/`MINOR` (now in `wspy.h`); bump it when fields are added/removed/renamed. This is the run *record*, not a run *configuration* â€” the lightweight config-file form of a run configuration is `wspy-run` (below); the full config-driven experiment-definition system remains a separate `INVESTIGATION.md`/4.3 item. `configuration_provenance` (`struct manifest_config_provenance`, `INVESTIGATION.md`'s "What shipped in 4.1", "structured configuration provenance") is a separate, below). `options.affinity` (`affinity.c`'s core/thread affinity control, `MANIFEST_SCHEMA_VERSION` `1.5.0` â†’ `1.6.0`) records the resolved `--affinity=<spec>` placement -- `requested` (the literal CLI text, `null` for the implicit `all` default), `mode` (`affinity_mode_name()`'s string), and `cpus` (the final resolved cpu list, `affinity_format_cpu_set()`) -- always populated, even for `all`, so a reader never has to cross-reference `host.num_cores` to know what it meant on that host. `topdown_formula_version` (top-level field, `MANIFEST_SCHEMA_VERSION` `1.6.0` â†’ `1.7.0`, `INVESTIGATION.md`'s 4.2 Tier 1 "Hierarchical topdown schema" item) records `TOPDOWN_FORMULA_VERSION` (`wspy.h`) â€” `null` when this run's `counter_mask` includes neither `COUNTER_TOPDOWN` nor `COUNTER_TOPDOWN2`, since the field describes a formula that wasn't exercised. `run_index.c`'s leaner record carries the identical field (also `RUN_INDEX_SCHEMA_VERSION` `1.6.0` â†’ `1.7.0`), same pairing convention as `counter_coverage`/`environment_coverage` above. `options.gpu` (`struct manifest_gpu_info`, `manifest.h`, `MANIFEST_SCHEMA_VERSION`/`RUN_INDEX_SCHEMA_VERSION` `1.7.0` â†’ device index (`null` if that vendor's flags weren't requested at all â€” gated on the `requested` flags themselves, not the index's sign, so a zero-initialized struct reads `null` rather than looking - `run_index.c`/`run_index.h` â€” appends one compact JSON Lines (JSONL) record per run to a shared file (`--run-index <file>`), independent of `--manifest` (a run can use either, neither, or both â€” if both are given, the index record's `output_files.manifest_path` points at the manifest file for that run). Built from the same `struct manifest_info` populated in `wspy.c:main()`, but versioned independently via `RUN_INDEX_SCHEMA_VERSION` since it's a leaner, line-oriented projection of a run, not the manifest itself. This is the "run index generation" piece of `INVESTIGATION.md`'s "Run artifact foundation" track (see "What shipped in 4.0") â€” it lets tooling query "all runs" by scanning one file instead of walking output directories. Appends are serialized with `flock(LOCK_EX)` so concurrent `wspy` processes sharing an index file don't interleave records. Carries the same `configuration_provenance` structured configuration provenance the manifest does (see `manifest.c`'s entry above), just in compact-JSON form.
- `json_util.c`/`json_util.h` â€” `json_write_string()` (escaping) and `format_iso8601()` (timestamp formatting) shared by `manifest.c` and `run_index.c`, the two hand-rolled JSON emitters in the tree.
- `json_reader.c`/`json_reader.h` â€” small recursive-descent JSON parser (`json_parse()`/`json_parse_file()` build a `struct json_value` tree; `json_free()`; `json_object_get()`/`json_array_get()`/`json_array_len()` plus `json_get_string()`/`json_get_number()`/`json_get_bool()` convenience accessors that return a default rather than asserting on a missing key or type mismatch). The read-side counterpart to `json_util.c`'s write helpers, and the first JSON *reader* in the tree â€” written for `validate.c`, not schema-specific to it.
- `validate.c` â€” `wspy-validate` binary: basic pre-publish quality checks against a `--manifest`-produced `manifest.json` (`INVESTIGATION.md`'s "Run artifact foundation" track, "Basic validation / quality checks pre-publish" item). Given one or more manifest paths, checks (independently, so one failure doesn't hide the others): the manifest's `schema_version` major version is one this build understands; every path in `output_files` exists; the `output` file is non-empty and, when `options.csv` is true, that every CSV row has the same column count as the header; `exit_status` reflects a clean exit (nonzero/signaled is a failure, `known: false` is a warning, mirroring `--exit-with-child`'s own "exit status not always known" caveat); `counter_coverage.measured < requested` is a warning, not a failure (partial coverage is `coverage.c`'s designed-in graceful degradation, not by itself a bad run); `timing.elapsed_seconds` is positive; and a small extensible `sanity_bounds[]` table (currently just `ipc`) plus a generic "numeric CSV column must be finite, non-negative, and not implausibly large" rule catch runaway/garbage values in known and unknown columns alike (percent-suffixed cells like `"26.61%"` get a looser bound since multi-core aggregate percentages legitimately exceed 100%). Reports each check as `[PASS]`/`[WARN]`/`[FAIL]`; exits 0 only if every manifest had zero `FAIL`s (`--strict` also fails on any `WARN`); `-q`/`--quiet` suppresses the full report for manifests with no `FAIL`. Scope is deliberately one manifest at a time, not a whole run index or output tree â€” that's the separate `ledger.c`/"coverage ledger" tool below.
- `ledger.c` â€” `wspy-ledger` binary: the "coverage ledger" item from `INVESTIGATION.md`'s "Run artifact foundation" track (workload status: done/skipped/unsupported/needs-tool-support, generated from the run index instead of hand-maintained). Not to be confused with `coverage.c`, which is per-run hardware *counter* coverage â€” this is suite-level *workload* coverage across many runs, scanned back out of one or more `--run-index` (JSONL) files via `json_reader.c`, the read side wspy-validate already introduced. Given a workload-list file (one workload name per line, optionally with a tab-separated `unsupported`/`needs-tool-support` annotation plus a free-text note) it matches each name as a substring against every run-index record's `command` array and infers a status: no matching record â†’ `skipped`; at least one matching record with a clean exit â†’ `done`; matching record(s) but none succeeded â†’ `needs-tool-support`. An explicit annotation in the workload list always wins over inference (for cases known in advance, e.g. a workload that needs a GPU-enabled build not available on this host, mirroring the "Intel not supported" early-exit `workload/cpu2017/run_test.sh` already hand-rolls). Prints a human report or `--csv`; `--strict` exits 1 if anything is `skipped`/`needs-tool-support`. `--unavailable-deps <file>` (a flat list of Phoronix Test Suite `ExternalDependencies` tags known unavailable on this host, e.g. `steam`, one per line) plus `--phoronix-profiles-dir <dir>` (default `$HOME/.phoronix-test-suite/test-profiles`) close the gap where a whole class of workloads shares one real-world blocker discovered by hand on just one of them (e.g. `dirt-rally2` failing because Steam can't be installed here) but every other Steam-dependent test would otherwise need the same discovery repeated one at a time: `scan_phoronix_dependencies()` reads each unannotated workload's own `<suite>/<test-id>-<version>/test-definition.xml` (matched by bare test id, i.e. the profile directory name with its trailing `-<version>` stripped) and, on a tag match, reports it as `unsupported` with a note naming the tag and profile â€” an explicit workload-list annotation or an already-successful run still both win over the scan, so this only ever fills in an inferred answer the workload list doesn't already have a better one for.
- `store.c` â€” `wspy-store` binary: SQLite-backed "canonical metrics schema + normalized store" item from `INVESTIGATION.md`'s "What shipped in 4.1" (both halves â€” a queryable run *catalog* and the per-run CSV *metric values* that make the catalog useful for stats/comparison work; see `doc/ARTIFACT_CONTRACT.md`'s "Normalized store" section for full detail). Given one or more `--run-index` (JSONL) files, ingests each record into a `runs` table (surrogate `id` PK, natural key `UNIQUE(hostname,run_id)` â€” `run_id` is only unique per host, per `run_index.c`'s own `format_run_id()` comment) plus `run_command_args`/`run_environment` child tables, all via `sqlite3_exec`/prepared statements against `json_reader.h`'s already-parsed tree (no separate schema-specific JSON walk). Idempotent: re-running against the same or a grown run-index file upserts (`ON CONFLICT(hostname,run_id) DO UPDATE`) rather than duplicating rows, and an `ingest_sources` table tracks each file's last-ingested byte offset so a growing file is only scanned incrementally, not re-parsed from byte 0 every time (falls back to a full rescan if the file shrank/rotated). A `(hostname,run_id)` collision â€” the same key with a materially different `start_time`/`command`, which can happen when containers share a host's UTS namespace with easily-duplicated low-numbered PIDs â€” is detected by comparing against the existing row before overwriting; on a mismatch the existing row is left alone, a warning is logged, and it's counted (visible via `--strict`) rather than silently merged. Each record's `output_files.manifest_path`, if present and the file exists, is also read (`json_parse_file()`) for the handful of host fields (`kernel_release`/`num_cores`/`num_cores_available`/`is_hybrid`) present in `manifest.c`'s `host{}` block but not in the leaner run-index record â€” cross-checked against the record's own `command[0]`/`start_time` before being trusted (a fixed/reused output filename could otherwise point at a different run's manifest), and skipped (not fatal) when the path is null, unreadable, or mismatched, matching this codebase's "measured vs unavailable" degrade-don't-fail idiom used throughout `coverage.c`/`provenance.c`. `ingest_csv_metrics()` does the same for `output_files.output_path` (when `options.csv` was true and the file exists and is non-empty): parses the CSV header/rows via `split_csv_line()`/`parse_numeric_field()` (mirrored verbatim from `validate.c`, same file format/rules) into the long/tall `metric_values` table â€” column *identity* always comes from the actual header row rather than being inferred from flags (a column named exactly `time`/`core`/`phase` is a dimension on every row emitted from that CSV row, every other non-empty-named column is a metric), which is what lets one code path cover the aggregate/`--interval`/`--per-core` shapes without special-casing any of them. A data row whose field count doesn't match the header's is skipped individually (counted, not fatal) rather than aborting the rest of the file â€” deliberately more robust than "stop at first mismatch," since it still recovers a `--interval --gpu-smi` run's correctly-shaped tail row despite every periodic tick being malformed (a confirmed pre-existing `wspy` bug â€” `timer_callback()` never emits the `gpu_smi_*` values a `--gpu-smi` CSV header promises), and correctly discards `--per-core --interval`'s separate unheaded trailing per-core block (a second confirmed pre-existing bug) without misattributing any of its values to the wrong metric name. `nan`/`-nan`/`inf` cells (which `topdown.c`'s ratio/percent `print_*` functions can legitimately emit on a zero divisor â€” confirmed empirically) are stored as `value = NULL` with `raw_text` always preserved for audit, so they're invisible to `AVG()`/`MIN()`/`MAX()` without losing the underlying signal. `preset_name`/`config_name` (from the record's `configuration_provenance.preset`/`.configuration`) and `affinity_mode`/`affinity_requested`/`affinity_cpus` (from `options.affinity`) are also bound directly onto the `runs` row â€” both were already present in every run-index record (`run_index.c`) but sat unused until `INVESTIGATION.md`'s "Comparison matrix mode deep-dive" made them queryable; the open-ended `configuration_provenance.options[]` array (`--config-option k=v` pairs â€” a wspy-sweep cell's own axis tag, or a uniform context label) goes into a new child table `run_config_options` via `replace_run_config_options()`, copying `replace_run_command_args()`'s own DELETE-then-INSERT idempotent-reingest shape line for line. `PRAGMA user_version` (not a table) gates schema compatibility, checked immediately after `sqlite3_open()`, with a real `ALTER TABLE`-based migration for existing v1 (pre-`metric_values`, see `MIGRATION_V1_TO_V2`) and v2 (pre-configuration-provenance/affinity, see `MIGRATION_V2_TO_V3`) databases; `--strict` exits 1 if any record was malformed, schema-major-mismatched, or collided. `#ifndef TEST_STORE` wraps only `usage()`/`main()`, matching `ledger.c`/`validate.c`, so `test_store.c` can `#include "store.c"` directly.
- `summary.c` â€” `wspy-summary` binary: the "summary table generator (min/max/median/mean/stddev/outlier flags) from indexed data" item from `INVESTIGATION.md`'s "What shipped in 4.1", closing the "a summary page can be regenerated from data only (no manual copy/paste)" criterion deferred from 4.0. Queries `store.c`'s normalized store (`--db <path>`, opened read-only â€” this tool never writes) directly via SQL rather than re-parsing run-index/CSV files itself, since `metric_values` already did the CSV-shape-independent parsing (aggregate/`--interval`/`--per-core` all land in the same long/tall table). For each run contributing to a `(group,metric)` bucket, `summarize()` first collapses that run's rows to one number via `AVG(value)` (a no-op for a single aggregate CSV row, the right collapse for `--interval` ticks or `--per-core` cores â€” the tool doesn't need to know which shape produced the data), then streams buckets (already sorted by the query's `ORDER BY`, so a bucket is always a contiguous run of rows) through `compute_stats()` for min/max/mean/median/`stddev` (sample, `n-1` denominator; reported as 0 for `n<2` rather than undefined) and a z-score outlier flag per value (`--outlier-stddev`, default 2.0 â€” requires `n>=3` and `stddev>0`, since flagging with fewer samples is meaningless). Every reported bucket additionally carries `cv_percent` (`stddev/|mean|`), a 95% confidence interval of the mean (`ci95_low`/`ci95_high`, `compute_ci95()` â€” Student's t via a small hardcoded `t95_table[30]` indexed by `df=n-1`, falling back to the normal approximation `z=1.96` past `df=30`; `n<2` degenerates to a zero-width interval at the mean without consulting the table, matching `stddev`'s own `n<2` convention) and a repeatability `verdict` (`compute_verdict()` â€” `PASS`, or `WARN:` plus a comma-separated, fixed-order reason list drawn from `thin`/`noisy`/`mixed-pmu`; "thin" is `n < 3`, reusing the outlier-flagging threshold above rather than a new number; "noisy" is `cv_percent` over `--max-cv`, default `5.0`, a single global threshold not per-metric) â€” all default output, no flag needed, `INVESTIGATION.md`'s "Repeatability policy + confidence metadata" item (see its "Repeatability policy + confidence metadata deep-dive" for the full design, including a documented caveat that a workload wrapping its own multi-trial benchmark harness â€” Phoronix's adaptive-N, SPEC's fixed multi-run high/low/median â€” can trigger `WARN:noisy` from the harness's own internal repeat-count variance as much as from real measurement noise, since this tool has no visibility into the wrapped child process and deliberately stays harness-agnostic rather than special-casing either one). `mixed-pmu` (`INVESTIGATION.md`'s 4.2 Tier 1, "PMU-capability-aware comparability warnings" item) is a fourth, independently-combinable verdict reason: a bucket's contributing runs are compared by `(cpu_vendor,counters_requested,counters_measured)` signature (`bucket_add()`'s `pmu_*` fields on `struct bucket`, from three columns `summarize()`'s query already selects off `runs` â€” no new store.c schema), and any deviation from the first-seen signature flags the whole bucket, exact-match rather than a numeric closeness threshold like `noisy`'s â€” there's no principled "how different is different enough" for a coverage triple the way there is for a CV percentage. "PMU capability" is deliberately `coverage.c`'s own vocabulary: different `cpu_vendor` means a same-named CSV column (e.g. `retire`) was likely computed from genuinely different raw hardware events (see `topdown.c`'s per-vendor formula tables); different `counters_requested` means the runs weren't even asking for the same counters (e.g. `--topdown` vs `--topdown2` both produce a `retire` column); different `counters_measured` with `counters_requested` held equal means one run's counter setup degraded (permission denial, multiplexing) while another's didn't â€” the cross-run aggregation blind spot `wspy-validate`'s own per-manifest partial-coverage warning can't see, since that tool never aggregates across runs. "Group" is workload command (`--group-by`'s default), `hostname`, `cpu_vendor`, `affinity_mode`/`preset_name`/`config_name` (plain `runs` columns since `INVESTIGATION.md`'s "Comparison matrix mode deep-dive"), or `cpu_governor`/`virt_role` (`run_environment` columns, always `LEFT JOIN`ed now, unconditionally â€” they'd been ingested since 4.0's `provenance.c` but had no join to reach them before this) â€” a fixed whitelist validated by `parse_group_by()` before being interpolated into the group-column position of the query (`group_by_column()` now returns a fully-qualified `r.`/`e.`-prefixed reference, the only place a `--group-by` value reaches raw SQL). `--group-by-option <name>` composes a *second*, independent grouping axis from an arbitrary `run_config_options.option_name` (a `--config-option` key, e.g. a `wspy-sweep` cell's own axis tag) â€” deliberately not folded into the fixed whitelist above, since it names a front-end-invented key rather than one of a known set; always passed as a bound query parameter (`LEFT JOIN run_config_options rco ON rco.run_id=r.id AND rco.option_name=?3`, `?3` bound to `""` when the flag isn't given, which never matches a real option name and so is a no-op rather than needing a second query shape). `struct bucket` carries this as `secondary_val` alongside `group_val`, comparing both when detecting a bucket boundary; the CSV/human output gains one new `group_by_option` column, present only when the flag was actually given (same conditional-column precedent as `--show-runs`' `contributing_runs`) â€” e.g. `wspy-summary --group-by command --group-by-option affinity` answers "for this workload, broken out by SMT on/off" rather than one flat regrouping. `--command`/`--hostname` filter which runs contribute (substring/exact match, bound as query parameters); `--metric` (repeatable) filters which metrics are reported, applied client-side after the query since SQLite parameter binding doesn't cover an `IN (...)` list cleanly. A bucket with fewer than `--min-runs` contributing runs is skipped (counted, not fatal â€” the same degrade-don't-fail idiom as `coverage.c`/`provenance.c`); `--strict` exits 1 if any bucket was skipped this way, if any reported bucket's `verdict` is `WARN`, or if nothing matched at all â€” the `WARN`-gates-`--strict` half matches `wspy-validate`'s own documented `--strict` behavior. `open_summary_db()` refuses a database whose `PRAGMA user_version` predates `store.c`'s `metric_values` table (schema version 2, `MIGRATION_V1_TO_V2`) but otherwise tolerates a newer schema version, since store.c's own versioning convention never removes/renames a column on a MINOR bump. `--show-runs`/`--trace` are `INVESTIGATION.md`'s "What shipped in 4.1", "Traceability links (summary row â†’ manifest â†’ raw CSV â†’ plots â†’ tree artifacts)" â€” the other criterion deferred from 4.0 alongside the summary generator above (see that doc's "Success criteria for a 4.0 kickoff"). `--show-runs` appends every contributing run's `hostname:run_id` to a bucket (all of them, not just `outlier_run_ids`' flagged subset, via the same `append_list_entry()` truncate-not-overflow helper `outlier_ids` now also shares) as a new trailing column, giving a surprising summary number a concrete run identity to chase; a bucket with enough runs to overflow the column's buffer gets a `(+N more)` marker rather than silently losing the tail, since (unlike `outlier_ids`, normally just a handful of flagged runs) a `--show-runs` list is meant to be complete. `--trace <hostname>:<run_id>` is a standalone mode (ignores every grouping/filter option, only needs `--db`) that resolves that identity via `trace_run()`, a direct `runs` table lookup by `(hostname,run_id)`, and prints the run's command/environment fields plus `stat()`/`opendir()`-checked existence for `store.c`'s recorded `manifest_path`/`output_path`/`tree_output_path` columns (one small loop over a `{path_key,exists_key,column}` table, not three copy-pasted blocks) and a derived `<output_path's dir>/plots` directory (`wspy-plot`'s output location, not a column store.c tracks anywhere; a bare relative `output_path` with no `/` at all has no derivable directory, so this degrades to `exists=0` rather than guessing a `plots` path relative to wherever `wspy-summary` happens to be running from) â€” closing the actual "manifest â†’ raw CSV â†’ plots â†’ tree artifacts" chain from a run identity, degrading each field independently (`exists=0`) rather than failing when a path no longer resolves (e.g. a run-index ingested from a different host, per `doc/ARTIFACT_CONTRACT.md`'s note that `manifest_path`/`output_path` are frequently host-relative in that setup). Output is stable `key=value` lines (embedded `\n`/`\r` in a value replaced with a space, since the format is one record per line), not `--csv`'s table shape or a bespoke JSON encoding, so a caller (`web/server.py`, a script) can parse a single resolved run without a JSON library on either side. Returns exit 1 (not 2) when the `(hostname,run_id)` pair isn't in the store at all, mirroring `--strict`'s "still needs more data, not a usage error" convention. `#ifndef TEST_SUMMARY` wraps only `usage()`/`main()`, matching `store.c`/`ledger.c`/`validate.c`, so `test_summary.c` can `#include "summary.c"` directly.
- `plot.c` â€” `wspy-plot` binary: the "Shared plotting templates" item from `INVESTIGATION.md`'s "What shipped in 4.1", replacing `workload/phoronix/gnuplot.sh` (retired) -- that script only knew two literal filenames (`amdtopdown.csv`/`systemtime.csv`) at fixed column positions; this generalizes to any wspy CSV output via the same normalized-schema convention `store.c`'s `ingest_csv_metrics()` established (a column named exactly `time`/`core`/`phase` is a dimension, every other non-empty-named column is a metric â€” column *identity*, not position or which flags produced it, decides what a CSV holds). - `preflight.c`/`preflight.h` â€” counter-fit preflight: estimates, from the counter groups a `counter_mask` would request, whether they'll fit in the available general-purpose hardware PMU counter slots without multiplexing, *before* any `perf_event_open()` calls are made (`INVESTIGATION.md`'s "Existing-capability extensions" track, "Counter-fit preflight" item â€” builds on availability/NMI-watchdog handling and `coverage.c` that already exist at runtime, surfacing the same fit information before a run instead of discovering it after, e.g. as silent undercounting since `read_counters()` doesn't yet scale by `time_running`/`time_enabled`). `preflight_evaluate_groups()` walks an already-built `counter_group_list` and sums, per group, only the counters that land on the per-core general-purpose PMU â€” `PERF_TYPE_HW_CACHE` groups count in full, `PERF_TYPE_RAW` groups count only entries whose per-counter `device_type` is itself `PERF_TYPE_RAW` (excluding AMD's L3 escape-hatch entries and AMD IBS's dynamic PMU types, both of which have their own separate uncore/system budget) â€” against an available-slot estimate (`nmi_running ? 5 : 6`, the same heuristic `topdown.c`'s AMD raw/cache group chunking already bakes in, kept in sync rather than inventing a second number). `preflight_evaluate()` is the convenience wrapper for real use, temporarily swapping the global `counter_mask` and calling `setup_counter_groups()` for a throwaway list (same idiom `wspy.c`'s `run_capabilities_probe()` uses for `--capabilities`) â€” needs no privileges and opens no perf events, since it's pure arithmetic over the event tables. `wspy.c:main()` calls it automatically right after `setup_raw_events()`, before any counter is opened, and `preflight_warn_if_tight()` prints a stderr warning (silent when the fit is fine, mirroring `check_nmi_watchdog()`'s style) with suggested downgrades (largest contributor(s) first, via a `COUNTER_*` bit â†’ `--no-<flag>` hint table) and, when the NMI watchdog is the reason a run doesn't fit, an explicit tip that stopping it (`scripts/setup_perf.sh` or `sudo sysctl -w kernel.nmi_watchdog=0`) frees 1 more slot. - `multipass.c`/`multipass.h` â€” the pure, unit-testable half of native multi-pass counter execution (`wspy --passes=<list>`, `INVESTIGATION.md`'s "What shipped in 4.1", "Native multi-pass counter execution"; the actual per-pass execution loop, `run_multipass()`, lives in `wspy.c` since it's coupled to that file's own execution-state globals â€” see `wspy.c`'s entry above). `multipass_group_names[]` maps each `--passes=<list>` token to a `COUNTER_*` bit, in ascending bit-value order (`wspy.h`'s own existing order) â€” that order is also the deterministic bin-packing iteration order, so `--passes=cache2,ipc` and `--passes=ipc,cache2` always produce the identical plan regardless of argv order. `multipass_plan_build()` greedily bin-packs the requested bits into passes that each fit the available general-purpose hardware PMU budget, reusing `preflight_evaluate()`'s real `setup_counter_groups()`-backed arithmetic at each step (not a hand-summed per-bit cost) â€” this correctly handles groups that share an underlying counter (e.g. `--dcache`/`--icache`/`--tlb` all reuse one `"instructions"` event in `cache_events[]`, `topdown.c`) rather than double-counting it. `COUNTER_SOFTWARE` is a deliberate special case: since `setup_counter_groups()` never builds a group for it itself (software counters are added separately, by both `main()`'s single-pass path and `run_multipass()`), it never registers any cost against the preflight budget â€” folded into whichever pass is already accumulating (or given a trivial pass of its own if none is open) rather than let a `preflight_evaluate()` call that can't see its true zero cost bounce it into a spurious extra re-execution of the workload. - `phase.c`/`phase.h` â€” interval automatic phase-boundary detection: classifies each `--interval` tick as `warmup`/`steady`/`degraded` from that tick's IPC (`INVESTIGATION.md`'s "Existing-capability extensions" track, "Interval (`--interval`) â†’ automatic phase-boundary detection" item â€” the basic-marker-detection half; named prerequisite for phase-aware topdown/IBS, 4.2). The single run-lifetime `phase_state` (`struct phase_detector`) is initialized once in `wspy.c:main()` right before CSV headers are written, then fed once per tick from both `topdown.c`'s `timer_callback()` and `wspy.c`'s own final (tail-interval) read, so the last CSV row/human print reflects the same classification a periodic tick would have. Not yet implemented: persisting phase boundaries into `--manifest`/`--run-index` â€” deferred until an actual consumer (phase-aware topdown, 4.2) needs them directly rather than via the CSV.
- `ibs.c`/`ibs.h` â€” capability-driven discovery of AMD's Instruction-Based Sampling (IBS) PMU support, plus the `ibs-basic`/`ibs-memory-deep` collection profiles built on top of it (`INVESTIGATION.md`'s "Zen5 / IBS" track). IBS is exposed as two dynamic PMUs the kernel registers at boot, `ibs_fetch`/`ibs_op`, present only when the CPU/kernel combination supports it; `ibs_probe()` reads `/sys/bus/event_source/devices/{ibs_fetch,ibs_op}/{type,format/*,caps/*}` at runtime (readdir-driven, not a hardcoded field list) rather than assuming which format fields (e.g. `l3missonly`, `ldlat`, `rand_en`) or caps (e.g. `zen4_ibs_extensions`, `dtlb_pgsize`) exist, since both vary by kernel version and CPU generation. Returns a `struct ibs_capabilities` (never fails; an absent PMU just yields `present`/`supported` 0) that `wspy --capabilities` prints via `print_ibs_capability_report()` alongside the existing counter capability report; `ibs_pmu_format()`/`ibs_pmu_cap()` are by-name lookup helpers into it.
  `--ibs-basic`/`--ibs-memory-deep` (`wspy.c`, setting `counter_mask |= COUNTER_IBS` and the module-global `ibs_collection_profile`) actually open IBS as counting events â€” `ibs_build_fetch_event()`/`ibs_build_op_event()` assemble a real `perf_event_attr` config by parsing each requested format field's sysfs `location` string (e.g. `"config1:0-11"`) into a (word, bit-range) and OR-ing the value in, rather than hardcoding bit offsets; `ibs_counter_group()` wires the result into an ordinary `struct counter_group` using the same `type_id == PERF_TYPE_RAW` + per-counter `device_type` escape hatch `raw_counter_group()` uses for L3 events (since `ibs_fetch`/`ibs_op` each have their own dynamic `type`, unlike a group's shared `type_id`). - `wspy-run` â€” bash profile-driven launcher wrapping `wspy`, not part of the C build (no compilation, just `chmod +x`). - `wspy-sweep` â€” Python declarative sweep runner (stdlib-only, `chmod +x`, no build step, same "thin client" principle as `wspy-queue`/`wspy-analyze`). `INVESTIGATION.md`'s 4.2 Tier 2 "Comparison matrix mode" item; see its "Comparison matrix mode deep-dive" for the full design. Cross-products `AXIS_HANDLERS` (v1: only `"affinity"` â€” SMT on/off, L3-domain placement, and core-type comparisons all fall out of the same `--affinity=<spec>` grammar `affinity.c` already defines) against one or more workload commands (`build_cells()`), invoking `wspy-run` once per cell, strictly sequentially (matching `wspy-queue`'s own one-PMU-at-a-time rule). Deliberately does not attempt to sweep the backlog line's other named axes (compiler/kernel/governor/VM-native) â€” the scope rule this tool holds itself to is "only automate axes that are process-scoped and side-effect-free outside the measured run" (`--affinity` is `sched_setaffinity()` on the forked child only; governor is a global sysfs write affecting every other process on the machine and outliving the run, a different, larger, not-yet-designed feature). An unrecognized `axes` key is a hard spec error (`fatal()`), not silently ignored, matching `--passes`' own fatal-on-unsupported-combination idiom. Each cell's own axis value(s) are tagged via `--config-option <axis-name>=<value>` (in addition to actually applying the axis, e.g. `--affinity`) specifically so `wspy-summary --group-by-option <axis-name>` can group on the *exact* swept value afterward â€” deliberately not relying on `summary.c`'s new `affinity_mode` column for this, since that's only the request's *category* (`"coretype"`) not which specific id was swept, which would merge distinct cells sweeping e.g. `coretype=0`/`coretype=1` into one bucket. Run-ids are `<sweep-invocation-timestamp>-<index>-<cell_id>` (`cell_id()` sanitizes workload name + sorted axis values into a filesystem-safe string) â€” a fresh timestamp per `wspy-sweep` invocation, not per cell, so re-running an identical spec later accumulates as new repeats in the store rather than colliding with (and silently updating) the prior run's identity. After the sweep (skipped entirely under `--dry-run`, which only prints every cell's command line), best-effort ingests into the normalized store (mirroring `web/joblib.py`'s `run_store_ingest_besteffort()`) and prints â€” never runs â€” the matching `wspy-summary --group-by command --group-by-option <axis>` command line, keeping execution and analysis as two separate, inspectable steps. A cell whose `wspy-run` invocation exits nonzero doesn't abort the sweep (the remaining cells still run), but is tracked and reported (with exit code) after the sweep finishes, and makes `wspy-sweep`'s own exit code nonzero. Coretype/domain axis values are host-specific (`affinity_topology_discover()`'s ascending-CPU scan order assigns the ids) â€” not portable across machines, so a spec sweeping them needs `wspy --list-affinity` run first to find real values for that host.

- `wspy-queue` â€” Python job/queue processor, alongside `wspy-run` (stdlib-only, `chmod +x`, no build step â€” same "thin client" principle as `web/server.py` below). This is `INVESTIGATION.md`'s "What shipped in 4.1" ("Deployment/hosting design note"), which landed as a real feature rather than staying design-only: a **job** is a portable, spec-only JSON document (`web/joblib.py`'s `build_job()`/`validate_job()`) capturing "what should run" â€” workload command, suite/benchmark identity, and either a `wspy-run` preset spec or the same checklist object `web/server.py`'s Run tab posts â€” captured *before* any run directory or output exists and carrying no reference to the machine that created it (no absolute `--output-root`, no path into that machine's `--run-index`/`store.db`), so a job file can be copied verbatim to a second machine with wspy checked out and processed there against that machine's own independent output tree. A job moves through a `pending` â†’ `running` â†’ `done`/`failed` lifecycle implemented as literal `<jobs-dir>/<state>/<job-id>.json` directories (Maildir-style); claiming a job for processing is one `os.rename()` from `pending/` to `running/`, atomic on a POSIX filesystem, which is also the only concurrency control needed since jobs are processed strictly serially (no `--parallel` â€” a `wspy` run has exclusive use of the machine's hardware PMU counters, matching `preflight.c`/`coverage.c`'s existing single-process-concern treatment of counter contention). A failed job stays in `failed/` until `wspy-queue requeue <job-id>` moves it back â€” no silent auto-retry, matching the degrade-don't-fail-but-don't-hide-it convention `coverage.c`/`validate.c` use elsewhere. Subcommands: `add` (creates a pending job â€” preset mode via `--profile`, custom mode via `--checklist-json <file>`, both also taking `--custom-plot`/`--only-custom`/toggle flags mirroring `wspy-plot`'s and the web UI's own vocabulary), `run` (processes pending jobs â€” `--job <id>` for one specific job, `--limit N`, `--strict` to exit 1 on any failure; drains everything currently pending by default, so a plain cron invocation works as a queue worker with no daemon/watch mode needed), `list`, `show <job-id>`, `requeue <job-id>`. All the actual command construction and execution (checklist/preset â†’ `wspy` argv, invoking `wspy`/`wspy-run`/`wspy-plot`/`wspy-store`, writing a `wspy-run`-shaped `manifest.json`/`summary.txt` for a custom-mode run) is shared with `web/server.py` via `web/joblib.py` (see that module's own docstring) rather than reimplemented â€” `wspy-queue` is a second front end over the same execution logic, headless and with no dependency on `web/server.py` being up. `web/server.py`'s Run tab is the primary job *creator* (a "Queue this instead of running it now" checkbox routes the same preset/checklist/workload state through `POST /api/enqueue-job` instead of launching immediately), while `wspy-queue` is the job *runner*; either tool can also do the other's job's own subset (`wspy-queue add` creates jobs too, for a fully browser-free workflow). Tested by `tests/wspy_queue_smoke.sh` (fake `wspy`/`wspy-run`/`wspy-plot`/`wspy-store` binaries â€” lifecycle, failure/requeue, and portability via copying a job file into a second, independent jobs/output tree) and `web/test_joblib.py` (job schema/validation and checklistâ†’argv logic, standalone `python3 web/test_joblib.py` â€” not wired into `make test`/`./run_tests.sh`, matching `web/`'s own "not covered" convention below).

Checking it under "System metrics" instead just folds `--power` into that system-wide pass, with no such side effect. Checking it in both places at once is allowed but redundant (measures power twice, across two separate re-executions of the workload) â€” nothing here reconciles that for the caller. The AMD IBS card's own `interval_secs` field (blank = aggregate, matching every other card's "Interval seconds" convention) is optional â€” unlike `wspy-run`'s `ibs-basic`/`ibs-memory-deep` builtin profiles (see that entry above), which now always pass `--interval 1`, since a preset is a fixed recipe rather than something a user tunes field by field. `build_configuration_passes(rundir, checklist)` is the one place checklist state becomes real `wspy` flags (whitelisted against `ALL_GROUPS`, the same name vocabulary `multipass.c`'s `multipass_group_names[]` uses for `--passes=<list>`) â€” reused identically by the preview endpoint and the real executor, so what's shown is never a paraphrase. Filters (`q` command substring, `suite`/`benchmark`/`hostname` substring, `cpu_vendor` exact, `status` exact, `date_from`/`date_to` against `start_time`'s date prefix) and pagination (`page`, `HISTORY_PAGE_SIZE`=25) are plain query-string GET parameters â€” bookmarkable/shareable, no JS, same pattern `/compare` already uses â€” and every field degrades independently (a run missing a manifest still lists with `â€”`/`unknown` rather than being dropped). Result rows carry the same "select and compare" checkboxes/`<form action="/compare">` the homepage table already uses, so a filtered search can feed directly into the curation studio's compare view.
  **The interactive tree viewer + run-to-run tree diff:** `GET /tree-viewer/<suite>/<benchmark>/<run_id>` (`render_tree_viewer()`) is a thin HTML shell over a new self-contained `web/static/proctree_viewer.js` (loaded only by this page and `/tree-diff`, rather than folded into the shared `app.js`, so no other page pays for this page-specific JS) â€” the actual collapsible tree/search-filter/column-toggle rendering happens entirely client-side, fetching the tree via `GET /api/tree-json/<suite>/<benchmark>/<run_id>` (`Handler._api_tree_json()`). That endpoint runs `proctree --json` against the run directory's `process.tree.txt` synchronously (`run_sync()`, same shape as the `_discovery_*` endpoints) and returns the parsed JSON directly â€” computed fresh on every request, no `.json` artifact written to disk (a deliberate design choice: always reflects the current tree file, never goes stale). Both `render_fixed_report()` and `render_wspy_run_report()` show an "Interactive tree viewer" link whenever `process.tree.txt` is present, right alongside wherever that file is already listed. `GET /tree-diff?r=<suite>/<benchmark>/<run_id>&r=...` (`render_tree_diff()`) is the run-to-run diff counterpart, reusing the homepage's/`/history`'s existing "select runs via checkbox" `<form method="get" action="/compare">` â€” a second submit button (`formaction="/tree-diff"`, "Tree diff selected") posts the identical selection to this new route instead, needing exactly two runs that both have a `process.tree.txt`. It fetches `GET /api/tree-diff-json?a=...&b=...` (`Handler._api_tree_diff_json()`), which runs `proctree --json` for each run into a `tempfile.NamedTemporaryFile` (cleaned up in a `finally`) and then `proctree --diff --json` against the two temp files, returning that merged diff tree for the same JS to render with added/removed/changed/same badges per node. `web/joblib.py`'s `build_proctree_json_argv()`/`build_proctree_diff_argv()` build these two commands' argv (alongside the pre-existing `build_proctree_argv()`); `parse_run_key()` (also `joblib.py`) factors out the `<suite>/<benchmark>/<run_id>` key-parsing both this feature and `render_compare()` need, so there's one implementation instead of two. See `proctree.c`'s own entry above for the `--json`/`--diff` CLI behavior these endpoints shell out to, and `doc/ARTIFACT_CONTRACT.md`'s "Tree JSON export"/"Tree diff JSON" sections for the exact schemas.
- `rocm/` â€” separate small C++ utilities (`smi_monitor`, `smi_info`) with their own Makefile, exploring the ROCm SMI API directly (not linked into `wspy`)
- `workload/` â€” driver scripts for external benchmark suites (SPEC CPU2017, pbbsbench, Phoronix) used to exercise wspy against real workloads; not part of the build. `workload/cpu2017/run_test.sh`, `workload/phoronix/run_test.sh`, and `workload/pbbsbench/run.sh` all call `wspy-run --suite <name> --benchmark <name>` (`deep-cpu,tree-heavy` on AMD, `deep-cpu-intel` on Intel â€” pbbsbench is AMD-only, no Intel branch) instead of hand-rolling the 7-8 `wspy` invocations they used to. See `wspy-run`'s own entry below for Phoronix `result_notifier` hook capture (`pts_hooks.log`) â€” that relocation now lives in `wspy-run` itself, not `workload/phoronix/run_test.sh`.
- `archive/wspy2.0/` â€” old version of the tool kept for reference, not built or maintained

**Key data structures** (`cpu_info.h`):
- `struct cpu_info` â€” root descriptor: vendor/family/model, core count, `systemwide_counters` or per-core counters
- `struct counter_group` â€” linked list of related counters (IPC, topdown, branch, cache, ...), chained via `.next`
- `struct counter_info` â€” one perf counter: fd, label, config, running/enabled times, `is_group_leader` flag
- `struct raw_event` â€” vendor-specific descriptor (`intel_raw_events[]`/`amd_raw_events[]` tables in `topdown.c`), parsed by `parse_intel_event()`/`parse_amd_event()` at startup into a `.raw.config` value via the `union intel_raw_cpu_format`/`union amd_raw_cpu_format` bitfields

**Counter mask bits** (`wspy.h`): `COUNTER_IPC`, `COUNTER_TOPDOWN[2]`, `COUNTER_TOPDOWN_{FE,BE,OP}`, `COUNTER_BRANCH`, `COUNTER_{D,I,L1,L2,L3}CACHE`, `COUNTER_MEMORY`, `COUNTER_TLB`, `COUNTER_OPCACHE`, `COUNTER_SOFTWARE`, `COUNTER_FLOAT` â€” combined into the global `counter_mask`, set via CLI flags in `parse_options()`. `system_mask` (`wspy.h`: `SYSTEM_LOADAVG|SYSTEM_CPU|SYSTEM_NETWORK` by default, `system.c`) is always populated; `--system`/`-s` (`sflag`) just controls whether `print_system()` is called, while `--gpu-busy`/`--gpu-metrics`/`--gpu-smi` additionally OR in `SYSTEM_GPU` regardless of `sflag`.

**Child launch protocol** (`topdown.c`): parent creates a pipe and forks; if `--tree` the child calls
`PTRACE_TRACEME` and blocks on the pipe; the parent sets up and starts counters, sleeps 2s, then writes
`"start\n"` to the pipe; the child reads it and `execve`s the real workload. With `--tree`, the parent
runs `ptrace_loop()` (setting `PTRACE_O_TRACEFORK|TRACEEXIT|TRACEEXEC`) to record fork/exec/exit events
with timestamps to the tree file; otherwise it just `wait4()`s on the child. `PTRACE_O_TRACEEXEC` is
what turns a traced process's routine post-`execve()` trap into a decodable `PTRACE_EVENT_EXEC` stop
(`"<time> <pid> exec <resolved /proc/<pid>/exe target>"`) instead of a bare, unclassifiable `SIGTRAP` --
without it, every exec anywhere in the tree previously fell through `ptrace_loop()`'s event switch into
the generic `"<time> <pid> unknown <status hex>"` catch-all (`status=0x57f`: `WIFSTOPPED`+`SIGTRAP` with
no event code in `status>>16`) â€” confirmed as the dominant source of `unknown_traps` in a real ~155K-
process build workload before this option was added.

**GPU support:** three independent GPU paths across two independent build axes (`AMDGPU=1`/`NVIDIA=1`,
either, both, or neither), all gated by `system_mask & SYSTEM_GPU` and only invoked when explicitly
requested via CLI flag:
- `--gpu-smi` (`AMDGPU=1`) â†’ `amd_smi.c` (ROCm `libamd_smi` library, linked at build time; includes `<amd_smi/amdsmi.h>`, so `ROCM_DIR/include/amd_smi/amdsmi.h` and `-lamd_smi` must exist under `ROCM_DIR`)
- `--gpu-busy` / `--gpu-metrics` (`AMDGPU=1`) â†’ `amd_sysfs.c` (direct sysfs reads: busy %, temp, activity, power, freq)
- `--gpu-nvidia` (`NVIDIA=1`) â†’ `nvidia_nvml.c` (NVML, `dlopen()`d at *run* time â€” no build-time header/library dependency; busy % + VRAM used/total, `nv_`-prefixed CSV columns so they coexist unambiguously with AMD's `gpu_*` columns when both `AMDGPU=1` and `NVIDIA=1` GPUs are active in the same run, e.g. an AMD iGPU + NVIDIA dGPU laptop)
When built without the matching `AMDGPU=1`/`NVIDIA=1`, these flags are still recognized by the option
parser but print a "GPU support not built" warning instead of erroring, so the binary behaves
consistently either way.

**CSV vs. human output:** every `print_*` function switches on `enum output_format` (`PRINT_NORMAL`,
`PRINT_CSV`, `PRINT_CSV_HEADER`). When adding a column, the header case and value case must be added
together and stay in the same order â€” `run_tests.sh` checks exact CSV column ordering (e.g.
`elapsed,utime,stime,gpu_busy,ipc`), so verify column position when adding new metrics. Two related
pitfalls `tests/golden_output.sh`'s generic column-count check (and manual review) should catch: the
`PRINT_CSV` value row must emit exactly the columns declared in `PRINT_CSV_HEADER` with **every field
comma-terminated** (a bare `fprintf(outfile,"%4.1f",x)` with no trailing comma silently fuses into the
next field instead of erroring), and it must do so **unconditionally** â€” gating the value row behind
`if (some_counter_value)` (rather than only gating on vendor/group applicability) means a
permission-denied run silently drops every column in that row while the header still claims them. Both
classes of bug shipped in this codebase before being caught by writing `tests/golden_output.sh` itself
(see that commit's history for the concrete cases: `--software --csv`, `--topdown-frontend`,
`--topdown-optlb`).

## Common edits

**New raw perf counter:** add an entry to `intel_raw_events[]`/`amd_raw_events[]` in `topdown.c` with an
`event=...,umask=...` description string and the appropriate `COUNTER_*` mask from `wspy.h` (add a new
bit there if needed); the parser converts the description to `.raw.config` in `setup_raw_events()`. Add
a CLI flag pair (`--foo`/`--no-foo`) in `wspy.c:parse_options()` if it should be user-toggleable.

**New counter group:** write a constructor in `topdown.c` following the existing pattern (e.g.
`software_counter_group()`, `cache_counter_group()`), allocate `struct counter_group` + `struct
counter_info[]`, and link it into the list in `setup_counter_groups()` based on `counter_mask`. Each
check there *prepends* to `*counter_group_list` (`cgroup->next = *counter_group_list; *counter_group_list
= cgroup;`), so groups print in the reverse of the order they're checked in `setup_counter_groups()` â€”
place a new `if (counter_mask & COUNTER_FOO)` block earlier/later in that function to control where it
lands in output, and check nearby groups' relative order if you need it to appear next to a specific one.

**New system metric:** parse it in `system.c:read_system()`, store it in `struct system_state`, print it
in `system.c:print_system()` with matching CSV/header/normal cases, and add a `SYSTEM_*` bit + CLI flag
if it should be independently toggleable.

**New manifest field:** add it to `struct manifest_info` in `manifest.h`, populate it at the call site in
`wspy.c:main()` (near the end, guarded by `if (manifest_path || run_index_path)`), and emit it in
`manifest.c:write_manifest()`. Adding a field is a backward-compatible change â€” bump the MINOR component
of `MANIFEST_SCHEMA_VERSION`; removing or renaming one is a MAJOR bump, since existing readers may depend
on the old shape.

**New run-index field:** most fields come from the same `struct manifest_info` used by the manifest (see
above), so if the field already exists there just emit it in `run_index.c:append_run_index()` too. A
field that's specific to the index record (not the manifest) needs its own plumbing through
`append_run_index()`'s signature. Either way, bump `RUN_INDEX_SCHEMA_VERSION` in `run_index.h` (MINOR for
an added field, MAJOR for removed/renamed) â€” it's versioned independently of `MANIFEST_SCHEMA_VERSION`.

**New normalized-store field (`store.c`):** add the column to the relevant `CREATE TABLE` in
`SCHEMA_DDL` (the fresh-database shape, already at the current version), bind it in
`upsert_run()`/`upsert_run_environment()`/`enrich_from_manifest()`/`ingest_csv_metrics()` as
appropriate (most run/environment fields come straight from the run-index record already parsed
via `json_reader.h`; manifest-only host fields go through `enrich_from_manifest()`; per-row CSV
values go through `ingest_csv_metrics()`), and bump `STORE_SCHEMA_VERSION` â€” checked via `PRAGMA
user_version` on open, not a `store_meta` row. Also add a matching `ALTER TABLE`/`CREATE TABLE IF
NOT EXISTS` block to a new `MIGRATION_V<N-1>_TO_V<N>` DDL string and a new branch in
`ensure_schema()`'s version dispatch (`user_version == 1` â†’ `MIGRATION_V1_TO_V2` is the existing
example to follow) â€” `ensure_schema()` runs *either* the fresh-DB `SCHEMA_DDL` (`user_version ==
0`) *or* exactly one migration step per existing older version, never both, so `ALTER TABLE ADD
COLUMN` never re-runs against a column that's already there. An old `wspy-store` binary still
refuses to write against a database whose `user_version` is newer than it understands, same as
before.

**New `wspy-summary --group-by` column (`summary.c`):** if it's a plain `runs` column, add a case to
`group_by_column()`'s switch (return the fully-qualified `r.<column>` reference) and a matching
`if (!strcmp(s,"<name>"))` line in `parse_group_by()`, plus a new `enum group_by` value â€” this is
still the fixed whitelist path, safe to interpolate directly into the query. If it's a
`run_environment` column instead, same shape but `e.<column>` (the query already unconditionally
`LEFT JOIN`s `run_environment`, so no query-shape change needed beyond the switch/enum). Don't add
an entry here for anything open-ended (a value not drawn from a small fixed set) â€” that's what
`--group-by-option`'s `run_config_options` join is for; extending the fixed whitelist to cover an
open-ended value would be the same class of mistake `parse_group_by()`'s own comment warns against
(it's what makes raw-SQL interpolation of the result safe in the first place).

**New `wspy-run` builtin profile:** add a `case` arm in `load_builtin_profile()` in `wspy-run` with a
`PASS_NAMES`/`PASS_FLAGS` array pair (one entry per pass, flags as a single space-separated string), and
add it to the `BUILTIN_PROFILES` list at the top of the file so `--list`/`--help` pick it up. Keep pass
names filesystem-safe (they become part of the output filename) and don't rely on flag ordering within a
pass beyond what `wspy`'s `getopt_long` already tolerates.

**New `wspy-validate` sanity bound:** add an entry to `sanity_bounds[]` in `validate.c` (column name plus
a `{min,max}` pair) when a specific CSV column deserves a tighter check than the generic "finite,
non-negative, not implausibly large" rule every numeric column already gets. Percent-suffixed cells
(e.g. `"26.61%"`) are matched by their trailing `%`, not by column name, and use the separate
`PERCENT_SANITY_MAX` bound instead of this table.

**New AMD IBS filter (`ibs.c`):** don't hardcode a bit offset â€” add the lookup/apply call in
`ibs_build_fetch_event()`/`ibs_build_op_event()` using `ibs_pmu_format(pmu,"<field name>")`, matching the
existing `l3missonly`/`ldlat`/`fetchlat` pattern: set a `*_requested` flag unconditionally, then only set
the matching `*_applied` flag and call `apply_format_field()` if the field lookup succeeds, so a
kernel/CPU that doesn't expose the field degrades to "unfiltered" instead of failing. If the filter
should be visible as a skew/quality annotation (most are, since any op/fetch filtering changes what
counts as a "sample"), surface it in `topdown.c`'s `print_ibs()` CSV/normal output too, not just as an
internal field on `struct ibs_event`.

## Notable runtime behavior

- NMI watchdog: if `/proc/sys/kernel/nmi_watchdog` is active, one hardware counter is reserved system-wide; `check_nmi_watchdog()` in `topdown.c` detects this and reduces available counter slots accordingly rather than failing outright.
- AMD L3 cache events depend on `/sys/devices/amd_l3/type` and are silently skipped if absent.
- Vendor detection distinguishes AMD Zen/Zen5, Intel Atom/Core (hybrid), and ARM systems; on ARM, `cpu_info.c` also tracks PMU cluster/type per core so `--per-core` can bind raw counters to the correct PMU type on mixed clusters.
- `amd_sysfs_initialize()` scans `/sys/class/drm/card*/device/vendor` for AMD-vendor (`0x1002`) cards via `amd_sysfs_scan_devices()`, rather than assuming a fixed `card1` path â€” on a multi-GPU machine (e.g. an NVIDIA card enumerated as `card1` ahead of the AMD card at `card2`) the old hardcoded path silently read nothing; the scan fixes that. If no AMD card is found, `--gpu-busy`/`--gpu-metrics` warn and read zero rather than erroring. `--gpu-device=<idx>` (recognized regardless of `AMDGPU` build, like the other `--gpu-*` flags) selects a specific card/device by index for both the sysfs (`--gpu-busy`/`--gpu-metrics`) and SMI (`--gpu-smi`) backends â€” omitted or negative, the prior lowest-numbered-card/device-0 default applies; `wspy --capabilities` prints both backends' enumerated device lists (index + selection) on `AMDGPU=1` builds so the indices to pass are discoverable up front.
- `amd_sysfs_gpu_metrics()` branches on the `gpu_metrics` binary blob's `format_revision` (1/2/3, structs in `amd_sysfs.h`) since the layout is GPU-generation-dependent; unrecognized revisions are logged and ignored (`gpu_metrics_state.valid` stays 0, so callers must check `amd_sysfs_gpu_metrics_valid()` before reading temp/activity/power/freq). See `doc/sysfs.gemini.txt` for background on the blob format and where the kernel defines it (`kgd_pp_interface.h`).
