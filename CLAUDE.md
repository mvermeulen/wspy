# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

wspy ("workload spy") is a C instrumentation wrapper: it launches a child workload as a subprocess and
collects runtime + hardware performance counter metrics around it (IPC, topdown, cache, TLB, branch,
software counters), plus optional system-wide (`/proc`, `/sys`) and AMD GPU metrics. Output is CSV or
human-readable. This is the author's internal research testbed, published for portability across
machines — treat it as a small, pragmatic C codebase rather than a polished library.

## Build & Test

- `make` — builds `wspy`, `cpu_info`, `proctree`, `wspy-validate`, `wspy-ledger`, `wspy-store`, `wspy-summary`, `wspy-plot` (default, no GPU support). `wspy-store`/`wspy-summary` link against the system SQLite library (`-lsqlite3`) — needs `libsqlite3-dev` (or your distro's equivalent) installed to build; `wspy`/the other tools (including `wspy-plot`) have no such requirement. `wspy-plot` shells out to a `gnuplot` binary at runtime (not a build-time dependency) — see its entry below.
- `make AMDGPU=1 [ROCM_DIR=/path/to/rocm]` — builds with AMD GPU support, also builds `amd_smi`, `amd_sysfs`. If `ROCM_DIR` isn't given, the Makefile auto-detects it by checking for `amd_smi/amdsmi.h` under `/opt/rocm` (the traditional ROCm install location) then `/usr` (where distro packages like Debian/Ubuntu's `rocm-smi-lib` install it), preferring `/opt/rocm` if both exist
- `make test` — builds and runs `test_wspy`, `test_proctree`, `test_validate`, `test_ledger`, `test_ibs`, `test_phase`, `test_store`, `test_summary`, and `test_plot` (unit-style tests; `test_wspy.c` `#include`s `wspy.c` directly with `TEST_WSPY`/`AMDGPU=0` defined to stub out `main()` and disable GPU code — the `test_wspy` Makefile target always compiles its own objects with `-DAMDGPU=0`, independent of whatever the top-level `AMDGPU` value is, so it can't pick up a mismatched GPU-enabled `topdown.o` from the same build tree; `test_validate.c` and `test_ledger.c` follow the same `#include "validate.c"`/`#include "ledger.c"` + `-DTEST_VALIDATE`/`-DTEST_LEDGER` pattern to reach each tool's static check functions; `test_ibs.c` similarly `#include`s `ibs.c` directly (no `TEST_IBS` macro needed since `ibs.c` has no `main()` to stub) and builds a fake `/tmp/test_ibs_sysfs/{ibs_fetch,ibs_op}/...` directory tree to exercise `ibs_probe_at()`'s sysfs-scanning logic without depending on real IBS hardware; `test_phase.c` follows the same direct-`#include "phase.c"` pattern, stubbing the handful of `wspy.h` globals (`interval`, `phase_flag`, `counter_mask`, `aflag`, `outfile`) and topdown.c's `find_ci_label()` it needs rather than linking the whole program, and drives `phase_detector_update()`'s state machine directly with synthetic IPC sequences to exercise warmup/steady/degraded transitions, hysteresis, and baseline drift without needing real perf counters; `test_store.c` follows the same direct-`#include "store.c"` + `-DTEST_STORE` pattern, using an in-memory (`:memory:`) SQLite database to exercise schema idempotency, upsert/re-ingest without duplication, `(hostname,run_id)` collision detection, manifest-enrichment matching/mismatch/missing-path handling, incremental byte-offset tracking on a growing run-index file, CSV metric-value ingestion (aggregate/`--interval`/`--per-core` shapes, percent and `nan`/`-nan` cell handling, per-row column-count-mismatch skipping, re-ingest idempotency, skip conditions) against synthetic CSV fixtures, and a hand-built v1-shaped database migrating cleanly to the current schema version via `ensure_schema()`, all without needing real `wspy` runs; `test_summary.c` follows the same direct-`#include "summary.c"` + `-DTEST_SUMMARY` pattern, using a minimal hand-built `:memory:` `runs`/`metric_values` fixture (not `store.c`'s full `SCHEMA_DDL`, to stay decoupled from that schema's own evolution) to exercise `compute_stats()`'s min/max/mean/median/stddev/outlier-flag math directly, `summarize()`'s per-run `AVG()` collapse and (group,metric)-bucket streaming against command/hostname/metric filters, `--min-runs` thin-bucket skipping, `cpu_vendor`-grouping's `NULL` → `"(unknown)"` bucket, `NULL`-only-metric exclusion, CSV field quoting, `--show-runs`' contributing-run-list column (present/absent, plus its `(+N more)` truncation marker under a large enough bucket), `trace_run()`'s artifact-path resolution against a real temp directory (existing files/dirs, stale/never-existed paths, a bare relative `output_path`'s plots-dir degradation, an unmatched `(hostname,run_id)`), and `open_summary_db()`'s schema-version gate against a real on-disk file (needed there specifically since it opens read-only, which a fresh `:memory:` connection can't meaningfully exercise); `test_plot.c` follows the same direct-`#include "plot.c"` + `-DTEST_PLOT` pattern, driving `match_templates()`/`add_fallback_match()`/`process_csv()` against synthetic CSV header strings and small on-disk CSV fixtures (amdtopdown-shaped, systemtime-shaped, an aggregate/no-`time`-column shape, an all-dimension shape, extra unclaimed columns) to exercise template matching, the generic fallback, and the "no time column"/"empty file" skip conditions -- entirely without gnuplot installed, since actually invoking gnuplot is only exercised by hand/in `run_tests.sh`'s environment, not this unit suite)
- `./run_tests.sh` — full test script: builds, runs unit tests, then integration-smoke-tests the real `wspy`/`proctree` binaries (CSV output, `--tree` output, network interface naming, GPU flag warnings when built without `AMDGPU=1`, `wspy-run` profile launching), runs `tests/golden_output.sh`, `tests/capability_matrix.sh`, `tests/wspy_queue_smoke.sh` (job/queue lifecycle tests against fake wspy/wspy-run/wspy-plot/wspy-store binaries — build/GPU-axis-independent, so it runs once, not per-build), and `tests/arm_topdown_microbench.sh` (the ARM microbench script self-skips on non-ARM hosts or without perf access), and additionally exercises the AMDGPU build if ROCm headers are found under `/opt/rocm` or `/usr` (re-running the golden-output/capability-matrix `tests/*.sh` scripts against that build too)
- `tests/golden_output.sh` — golden output-contract tests (`INVESTIGATION_4.0.md`'s "Testing and documentation" track): pins the current, intentional shape of `wspy`'s output without needing root/perf access — exact-match CSV headers for the base/rusage columns and, vendor-gated via `./cpu_info`, every AMD raw-event counter group plus a small native multi-pass counter execution (`--passes=ipc,cache2`) bundle pinning the merged-row column order; a regex check for `--system`'s host-specific network-interface columns; a vendor-neutral generic check (CSV header/value column counts must match) run across the full flag matrix; a handful of always-present normal-output summary-line checks; and a `--tree` line-grammar/`ptrace-summary`-footer consistency check at a small, readable scale (`run_tests.sh` keeps its own larger 2000-process version as a stress/integrity test, not a format-contract one). Can be run standalone once `wspy`/`cpu_info` are built.
- `tests/capability_matrix.sh` — capability-matrix smoke tests (same `INVESTIGATION_4.0.md` track): runs ~35 flag-bundle combinations (one per counter group, GPU flags, per-core/interval/tree modifiers, manifest/run-index/`--capabilities`, a kitchen-sink combo, native multi-pass counter execution (`--passes`), and `--exit-with-child`'s two expected exit codes) and asserts each exits as expected, never prints `fatal error`, and shows no crash indication — the graceful-degradation contract. `--passes`' own incompatibility checks (`--interval`/`--per-core`/`--tree`/`--ibs-basic` combined with `--passes`) are the one deliberate exception, asserted via a separate `run_expected_fatal_bundle()` helper that requires a fatal error rather than forbidding one, since these bundles are supposed to be rejected outright. Auto-detects CPU vendor (`./cpu_info`) and GPU build (whether `--gpu-busy` warns "GPU support not built") and reports both, since neither axis is switchable from a single host/build; `run_tests.sh` calls it once per build it produces so both ends of the GPU-build axis get covered in one pass. Known gap (documented in-script, not fixed): `--per-core` combined with any counter group produces a CSV header with only the base/coverage columns while each per-core row still appends that group's values, so the `per-core-topdown` bundle deliberately doesn't assert CSV column-count matching — see the comment there.
- `./test_amd_smi.sh` — focused smoke test for the `amd_smi` module (requires `AMDGPU=1` build)
- `make clean` — remove `.o`/backup files; `make clobber` — also remove built binaries
- No root required for `./cpu_info` (CPU detection only). Perf counters and `--tree` (ptrace) require root, or `CAP_SYS_PTRACE` + `perf_event_paranoid <= 1` — use `scripts/setup_perf.sh` to check/adjust `nmi_watchdog` and `perf_event_paranoid` sysctls.

Quick manual checks:
```
./cpu_info                          # CPU/core detection, no privileges needed
sudo ./wspy -- sleep 1              # basic run, default IPC counters
sudo ./wspy --csv --topdown -- true # CSV output with topdown metrics
```

Web launcher/report browser (`web/server.py`, see its `CLAUDE.md` entry below): `python3 web/server.py`,
then open `http://127.0.0.1:8765/`. Stdlib-only, no build step or dependency; not covered by `make`/
`./run_tests.sh`.

## Development workflow

Starting with the 4.0 development cycle (see `INVESTIGATION_4.0.md`), new feature work happens on a
short-lived feature branch rather than committing directly to `master`. Small housekeeping changes
(typo fixes, doc tweaks, README updates) can still go straight to `master` — this applies to actual
feature/behavior changes, especially anything tracked in `INVESTIGATION_4.0.md`'s phase-tagged
priority lists.

- **Branch naming:** `feature/<slug>`, e.g. `feature/run-manifest`, `feature/gpu-path-scan`. Prefer
  one branch per inventory row/idea in `INVESTIGATION_4.0.md` rather than one branch per phase, so
  each PR stays reviewable and revertable independently.
- **Starting a feature:** `git checkout master && git pull && git checkout -b feature/<slug>`.
- **While working:** commit normally on the branch; rebase or merge `master` into the branch as
  needed to stay current, but don't rewrite history that's already been pushed.
- **Finishing a feature:** push the branch (`git push -u origin feature/<slug>`) and open a PR with
  `gh pr create` (origin is already `github.com/mvermeulen/wspy`). Merge through GitHub once it's
  ready — don't merge feature branches into `master` locally and push `master` directly.
- **Before opening the PR:** run `./run_tests.sh` (and `./test_amd_smi.sh` if the change touches GPU
  code) — the branch should be in a state that passes the existing test suite. `run_tests.sh` now
  includes `tests/golden_output.sh`'s CSV column-order/summary-fragment/tree-format checks and
  `tests/capability_matrix.sh`'s flag-bundle graceful-degradation checks (see "Build & Test" above),
  so most "CSV vs. human output" contract regressions (see below) are caught automatically; extend
  those two files' tables when a change adds/reorders/removes a column rather than relying on manual
  checking alone.
- **Scope:** keep a feature branch to one inventory idea where practical; large phase-sized efforts
  (e.g. all of "Run artifact foundation") should still land as a series of small merged PRs rather
  than one long-lived branch, to avoid the drift and merge pain of a big-bang branch.

## Architecture

**Data flow:** CLI (`wspy.c`) → CPU detection (`cpu_info.c`) → counter setup (`topdown.c`) → fork/exec
child with optional ptrace → periodic/final counter reads → CSV or human-readable output.

**Files and responsibilities:**
- `wspy.c` — `main()`, `parse_options()` (long/short option parsing into globals like `counter_mask`, `system_mask`, `aflag`/`sflag`/`csvflag`/`treeflag`), orchestrates setup → start → read → print lifecycle. `--passes=<list>` (native multi-pass counter execution, `multipass.h`/`multipass.c` — `INVESTIGATION_4.0.md`'s 4.1 Tier 1 item) branches `main()` into `run_multipass()` instead: re-launches the workload once per automatically-sized pass from `multipass_plan_build()`, merging the result into one CSV/manifest/run-index record rather than requiring N separate `wspy` invocations the way `wspy-run`'s builtin profiles do today (see `wspy-run`'s own entry below — it isn't changed by this, `--passes` is a lower-level capability it could adopt later). Each pass gets its own `setup_counter_groups()`/`setup_counters()`/`start_counters()`/`launch_child()`/`wait4()`/`read_counters()` cycle exactly like the single-pass path, but MUST `close_counters()` (`topdown.c`) its fds before the next pass's `setup_counters()` opens new ones — nothing in this codebase closed a counter fd before this feature existed, since a single-pass run just exits with its fds open. `print_metrics()` needs no changes: every counter group is self-normalizing (each raw/cache group carries its own denominator counters, per `find_ci_label()`'s group-scoped lookups) and dispatches purely off `cgroup->mask`, so calling it once per pass into the same row just concatenates that pass's columns, with no possible name collision since the bin-packer never duplicates a `COUNTER_*` bit across passes. Because each pass is a genuinely separate re-execution of the workload (its own start/finish time, rusage, exit status), there's no single canonical value for those across N runs -- pass 0 ("primary") supplies the merged row's base columns and the manifest's top-level `exit_status`, while every pass's own timing/exit status/coverage delta is still recorded in the manifest's `passes[]` array for full audit (a differing exit status across passes warns, doesn't fail the run). V1 scope is aggregate-only: `--interval`, `--per-core`, `--tree`, `--ibs-basic`/`--ibs-memory-deep`, and the GPU flags are all fatal'd against `--passes` (checked right after `parse_options()` returns), since each either has no defined multi-pass merge semantics or already has its own separate collection/budget model. `populate_manifest_common()`/`exit_code_epilogue()` factor the manifest-population/`--exit-with-child` epilogue logic shared between `main()`'s single-pass tail and `run_multipass()`'s tail, so both stay in sync. `--multiplex` (opt-in, off by default) swaps `run_multipass()`'s plan builder from `multipass_plan_build()` to `multipass_plan_build_multiplexed()`: instead of bin-packing the requested groups into N passes that each fit the general-purpose PMU budget, it opens every requested group in one pass and lets the kernel multiplex whatever doesn't fit, trading precision (heavier multiplexing means lower per-counter `confidence_ratio()`) for a single workload execution — safe to offer now that `read_counters()`'s multiplex scaling (see its entry below) keeps an oversubscribed run's *values* correct, not just flagged as low-confidence. `--multiplex` given without `--passes` is fatal, since it has no meaning on its own. `--preset-name <name>`/`--config-name <name>`/`--config-option <k>=<v>` (repeatable) are purely metadata (`INVESTIGATION_4.0.md` item 16, "structured configuration provenance") populating `manifest_info.config_provenance` (`manifest.h`) — `wspy` itself never reads them back or lets them affect the run, they only exist so a front end that *does* have a preset/configuration/option vocabulary (`wspy-run`, the web launcher) can record its own choice of words into the manifest/run-index rather than a report having to re-derive "what this was" from `counter_mask`/argv. A malformed `--config-option` (no `=`) is warned about and dropped, not fatal.
- `topdown.c` (largest file, ~68K) — vendor event tables (`intel_raw_events[]`, `amd_raw_events[]`, `arm_raw_events[]`), `perf_event_open` wrapper, counter group constructors, `setup_counters`/`start_counters`/`read_counters`, all `print_*` formatters, `launch_child`/`ptrace_setup`/`ptrace_loop` (child launch + fork/exec/exit tracing), `check_nmi_watchdog`. ARM now has a topdown-equivalent decomposition wired through raw PMU events (`op_retired`/`op_spec`/`stall_slot*`/`stall_backend_mem`/branch events) rather than perf's metricgroup interface, and `setup_counters()` honors per-core `target_cpu` binding in `--per-core` mode so mixed ARM PMU clusters can route RAW events to each core's own PMU type. `ptrace_loop()`'s syscall-stop handling reads the traced process's registers through `ptrace_arch.h`'s `wspy_regs_t`/`ptrace_getregs()`/`PTRACE_SYSCALL_NUM`/`PTRACE_SYSCALL_ARG2` rather than `struct user_regs_struct`/`PTRACE_GETREGS`/`.orig_rax`/`.rsi` directly, so it isn't x86_64-hardcoded (see `ptrace_arch.h` below). `read_counters()` scales every counter's raw delta by that read's own multiplex ratio (`time_enabled`-delta/`time_running`-delta since the previous read, tracked via `struct counter_info`'s `last_time_running`/`last_time_enabled`, `cpu_info.h`) before storing it in `.value` (`INVESTIGATION_4.0.md` 4.1 Tier 1 "Correctness" item) — a counter multiplexed off the PMU for part of a window previously reported its raw (undercounted) count with only the separate confidence envelope hinting at the loss; now the value itself is extrapolated to the full window, and `time_running`/`time_enabled` on `struct counter_info` hold that read's *delta* (not cumulative-since-start) for `confidence_ratio()`/`phase_current_ipc()` to gate on. `print_ipc()` and `phase.c`'s `phase_current_ipc()` used to redo this scaling themselves from raw `cpu-cycles`/`instructions` values; both were simplified to read the now-pre-scaled `.value` directly, since rescaling there would double-count it.
- `ptrace_arch.h` — architecture-neutral ptrace register access: `wspy_regs_t` (typedef for `struct user_regs_struct`, whose layout differs per arch despite the shared name), `ptrace_getregs(pid,regs)`, and the `PTRACE_SYSCALL_NUM(regs)`/`PTRACE_SYSCALL_ARG2(regs)` macros `ptrace_loop()` (`topdown.c`) uses for the syscall number and `SYS_openat`'s pathname-pointer argument. The `INVESTIGATION_4.0.md` "Arch-neutral ptrace register-access macros" item — a mechanical refactor of existing x86_64 logic (`PTRACE_GETREGS`/`.orig_rax`/`.rsi`) behind `#ifdef __x86_64__`, done ahead of an actual ARM64 backend so the abstraction doesn't need retrofitting once more logic depends on the current shape. The `#elif defined(__aarch64__)` branch (`PTRACE_GETREGSET`+`NT_PRSTATUS`, syscall number/arg2 from `regs[8]`/`regs[1]`) is modeled on documented arm64 ptrace/syscall-ABI conventions but has never been built or run on real ARM64 hardware — unverified prep, not a working ARM64 backend. Building for any other architecture fails at compile time (`#error`), same as `cpu_info.c`'s `<cpuid.h>` use already does.
- `cpu_info.c`/`cpu_info.h` — vendor/model detection (`inventory_cpu()`), core enumeration, hybrid-CPU handling (Intel Atom+Core mixes), ARM fallback topology inventory, and ARM PMU-cluster discovery from sysfs (`armv8_pmuv3_*` `cpus` maps); defines `struct cpu_info`, `struct cpu_core_info`, the Intel/AMD raw perf-event format unions, and `struct counter_group`/`struct counter_info`
- `system.c` — system-wide metrics: load average, `/proc/stat` CPU time, `/proc/net/dev` network counters, AMD GPU busy % (if `AMDGPU=1`); `read_system()`/`print_system()`
- `amd_smi.c`/`amd_smi.h` — GPU metrics via ROCm's `libamd_smi` (only compiled with `AMDGPU=1`). `amd_smi_initialize(int device_index)` enumerates every socket/device via `amdsmi_get_socket_handles()`/`amdsmi_get_processor_handles()` as before, then selects which enumerated device `amd_smi_metrics()`/`amd_smi_memory()` read from: `-1` picks device 0 (prior default), otherwise the given index must be `< amd_smi_device_count()` or it's logged and no device is selected (metrics/memory stay invalid rather than silently reading a different device than requested). `amd_smi_print_capability_report()` lists every enumerated device with its owning socket name and whether it's the selected one.
- `amd_sysfs.c`/`amd_sysfs.h` — GPU metrics (busy %, temp, activity, power, freq) read directly from sysfs, no ROCm dependency required at this layer but still gated by `AMDGPU=1`. `amd_sysfs_scan_devices()` enumerates every AMD (vendor `0x1002`) card under `/sys/class/drm`, ascending by card index; `amd_sysfs_initialize(int device_index)` takes `-1` for the prior lowest-numbered-card default or an explicit index that must exactly match a scanned card, else it degrades to no sysfs GPU data (logged, not fatal). `amd_sysfs_print_capability_report()` lists every scanned card and marks the selected one.
- `proctree.c` — standalone `proctree` binary that parses the tree file produced by `wspy --tree` (not CSV: one of four line kinds — `<time> root <pid>`, `<time> start <pid> <ppid>`, `<time> exit <pid> <stat-fields>`, `<time> comm <pid> <name>` — see the comment at the top of `proctree.c`) and reconstructs the process hierarchy
- `error.c`/`error.h` — centralized logging: `fatal()`, `error()`, `warning()`, `notice()`, `debug()`, `debug2()`, gated by `set_error_level()`
- `manifest.c`/`manifest.h` — writes the optional JSON run manifest (`--manifest <file>`): command line, start/finish timestamps, child exit status (when known — not available in `--tree` mode since `ptrace_loop` reaps children itself), host/CPU info, the option flags used, and the list of output files produced. `MANIFEST_SCHEMA_VERSION` in `manifest.h` is the SemVer of the manifest's JSON *shape*, independent of `WSPY_VERSION_MAJOR`/`MINOR` (now in `wspy.h`); bump it when fields are added/removed/renamed. This is the run *record*, not a run *configuration* — the lightweight config-file form of a run configuration is `wspy-run` (below); the full config-driven experiment-definition system remains a separate `INVESTIGATION_4.0.md`/4.3 item. `configuration_provenance` (`struct manifest_config_provenance`, `INVESTIGATION_4.0.md` item 16 "structured configuration provenance") is a separate, narrower piece of this: which named preset (if any) and/or launcher-vocabulary configuration category/options produced this run, in the same words a front end's own preset/checklist UI uses — `wspy` itself has no notion of presets/configurations, so this is populated purely from the metadata-only `--preset-name`/`--config-name`/`--config-option <k>=<v>` CLI flags (`wspy.c`), all-NULL/`[]` (not a gap) for a plain direct invocation that doesn't pass them. `wspy-run`'s `run_pass()` and `web/joblib.py`'s `build_pass_argv()` are the two front ends that populate it today (see their own entries below).
- `run_index.c`/`run_index.h` — appends one compact JSON Lines (JSONL) record per run to a shared file (`--run-index <file>`), independent of `--manifest` (a run can use either, neither, or both — if both are given, the index record's `output_files.manifest_path` points at the manifest file for that run). Built from the same `struct manifest_info` populated in `wspy.c:main()`, but versioned independently via `RUN_INDEX_SCHEMA_VERSION` since it's a leaner, line-oriented projection of a run, not the manifest itself. This is the "run index generation" piece of `INVESTIGATION_4.0.md`'s "Run artifact foundation" track (see "What shipped in 4.0") — it lets tooling query "all runs" by scanning one file instead of walking output directories. Appends are serialized with `flock(LOCK_EX)` so concurrent `wspy` processes sharing an index file don't interleave records. Carries the same `configuration_provenance` structured configuration provenance the manifest does (see `manifest.c`'s entry above), just in compact-JSON form.
- `json_util.c`/`json_util.h` — `json_write_string()` (escaping) and `format_iso8601()` (timestamp formatting) shared by `manifest.c` and `run_index.c`, the two hand-rolled JSON emitters in the tree.
- `json_reader.c`/`json_reader.h` — small recursive-descent JSON parser (`json_parse()`/`json_parse_file()` build a `struct json_value` tree; `json_free()`; `json_object_get()`/`json_array_get()`/`json_array_len()` plus `json_get_string()`/`json_get_number()`/`json_get_bool()` convenience accessors that return a default rather than asserting on a missing key or type mismatch). The read-side counterpart to `json_util.c`'s write helpers, and the first JSON *reader* in the tree — written for `validate.c`, not schema-specific to it.
- `validate.c` — `wspy-validate` binary: basic pre-publish quality checks against a `--manifest`-produced `manifest.json` (`INVESTIGATION_4.0.md`'s "Run artifact foundation" track, "Basic validation / quality checks pre-publish" item). Given one or more manifest paths, checks (independently, so one failure doesn't hide the others): the manifest's `schema_version` major version is one this build understands; every path in `output_files` exists; the `output` file is non-empty and, when `options.csv` is true, that every CSV row has the same column count as the header; `exit_status` reflects a clean exit (nonzero/signaled is a failure, `known: false` is a warning, mirroring `--exit-with-child`'s own "exit status not always known" caveat); `counter_coverage.measured < requested` is a warning, not a failure (partial coverage is `coverage.c`'s designed-in graceful degradation, not by itself a bad run); `timing.elapsed_seconds` is positive; and a small extensible `sanity_bounds[]` table (currently just `ipc`) plus a generic "numeric CSV column must be finite, non-negative, and not implausibly large" rule catch runaway/garbage values in known and unknown columns alike (percent-suffixed cells like `"26.61%"` get a looser bound since multi-core aggregate percentages legitimately exceed 100%). Reports each check as `[PASS]`/`[WARN]`/`[FAIL]`; exits 0 only if every manifest had zero `FAIL`s (`--strict` also fails on any `WARN`); `-q`/`--quiet` suppresses the full report for manifests with no `FAIL`. Scope is deliberately one manifest at a time, not a whole run index or output tree — that's the separate `ledger.c`/"coverage ledger" tool below.
- `ledger.c` — `wspy-ledger` binary: the "coverage ledger" item from `INVESTIGATION_4.0.md`'s "Run artifact foundation" track (workload status: done/skipped/unsupported/needs-tool-support, generated from the run index instead of hand-maintained). Not to be confused with `coverage.c`, which is per-run hardware *counter* coverage — this is suite-level *workload* coverage across many runs, scanned back out of one or more `--run-index` (JSONL) files via `json_reader.c`, the read side wspy-validate already introduced. Given a workload-list file (one workload name per line, optionally with a tab-separated `unsupported`/`needs-tool-support` annotation plus a free-text note) it matches each name as a substring against every run-index record's `command` array and infers a status: no matching record → `skipped`; at least one matching record with a clean exit → `done`; matching record(s) but none succeeded → `needs-tool-support`. An explicit annotation in the workload list always wins over inference (for cases known in advance, e.g. a workload that needs a GPU-enabled build not available on this host, mirroring the "Intel not supported" early-exit `workload/cpu2017/run_test.sh` already hand-rolls). Prints a human report or `--csv`; `--strict` exits 1 if anything is `skipped`/`needs-tool-support`.
- `store.c` — `wspy-store` binary: SQLite-backed "canonical metrics schema + normalized store" item from `INVESTIGATION_4.0.md`'s 4.1 Tier 1 list (both halves — a queryable run *catalog* and the per-run CSV *metric values* that make the catalog useful for stats/comparison work; see `doc/ARTIFACT_CONTRACT.md`'s "Normalized store" section for full detail). Given one or more `--run-index` (JSONL) files, ingests each record into a `runs` table (surrogate `id` PK, natural key `UNIQUE(hostname,run_id)` — `run_id` is only unique per host, per `run_index.c`'s own `format_run_id()` comment) plus `run_command_args`/`run_environment` child tables, all via `sqlite3_exec`/prepared statements against `json_reader.h`'s already-parsed tree (no separate schema-specific JSON walk). Idempotent: re-running against the same or a grown run-index file upserts (`ON CONFLICT(hostname,run_id) DO UPDATE`) rather than duplicating rows, and an `ingest_sources` table tracks each file's last-ingested byte offset so a growing file is only scanned incrementally, not re-parsed from byte 0 every time (falls back to a full rescan if the file shrank/rotated). A `(hostname,run_id)` collision — the same key with a materially different `start_time`/`command`, which can happen when containers share a host's UTS namespace with easily-duplicated low-numbered PIDs — is detected by comparing against the existing row before overwriting; on a mismatch the existing row is left alone, a warning is logged, and it's counted (visible via `--strict`) rather than silently merged. Each record's `output_files.manifest_path`, if present and the file exists, is also read (`json_parse_file()`) for the handful of host fields (`kernel_release`/`num_cores`/`num_cores_available`/`is_hybrid`) present in `manifest.c`'s `host{}` block but not in the leaner run-index record — cross-checked against the record's own `command[0]`/`start_time` before being trusted (a fixed/reused output filename could otherwise point at a different run's manifest), and skipped (not fatal) when the path is null, unreadable, or mismatched, matching this codebase's "measured vs unavailable" degrade-don't-fail idiom used throughout `coverage.c`/`provenance.c`. `ingest_csv_metrics()` does the same for `output_files.output_path` (when `options.csv` was true and the file exists and is non-empty): parses the CSV header/rows via `split_csv_line()`/`parse_numeric_field()` (mirrored verbatim from `validate.c`, same file format/rules) into the long/tall `metric_values` table — column *identity* always comes from the actual header row rather than being inferred from flags (a column named exactly `time`/`core`/`phase` is a dimension on every row emitted from that CSV row, every other non-empty-named column is a metric), which is what lets one code path cover the aggregate/`--interval`/`--per-core` shapes without special-casing any of them. A data row whose field count doesn't match the header's is skipped individually (counted, not fatal) rather than aborting the rest of the file — deliberately more robust than "stop at first mismatch," since it still recovers a `--interval --gpu-smi` run's correctly-shaped tail row despite every periodic tick being malformed (a confirmed pre-existing `wspy` bug — `timer_callback()` never emits the `gpu_smi_*` values a `--gpu-smi` CSV header promises), and correctly discards `--per-core --interval`'s separate unheaded trailing per-core block (a second confirmed pre-existing bug) without misattributing any of its values to the wrong metric name. `nan`/`-nan`/`inf` cells (which `topdown.c`'s ratio/percent `print_*` functions can legitimately emit on a zero divisor — confirmed empirically) are stored as `value = NULL` with `raw_text` always preserved for audit, so they're invisible to `AVG()`/`MIN()`/`MAX()` without losing the underlying signal. `PRAGMA user_version` (not a table) gates schema compatibility, checked immediately after `sqlite3_open()`, with a real `ALTER TABLE`-based migration for existing v1 (pre-`metric_values`) databases (see `MIGRATION_V1_TO_V2`); `--strict` exits 1 if any record was malformed, schema-major-mismatched, or collided. `#ifndef TEST_STORE` wraps only `usage()`/`main()`, matching `ledger.c`/`validate.c`, so `test_store.c` can `#include "store.c"` directly.
- `summary.c` — `wspy-summary` binary: the "summary table generator (min/max/median/mean/stddev/outlier flags) from indexed data" item from `INVESTIGATION_4.0.md`'s 4.1 Tier 1 list, closing the "a summary page can be regenerated from data only (no manual copy/paste)" criterion deferred from 4.0. Queries `store.c`'s normalized store (`--db <path>`, opened read-only — this tool never writes) directly via SQL rather than re-parsing run-index/CSV files itself, since `metric_values` already did the CSV-shape-independent parsing (aggregate/`--interval`/`--per-core` all land in the same long/tall table). For each run contributing to a `(group,metric)` bucket, `summarize()` first collapses that run's rows to one number via `AVG(value)` (a no-op for a single aggregate CSV row, the right collapse for `--interval` ticks or `--per-core` cores — the tool doesn't need to know which shape produced the data), then streams buckets (already sorted by the query's `ORDER BY`, so a bucket is always a contiguous run of rows) through `compute_stats()` for min/max/mean/median/`stddev` (sample, `n-1` denominator; reported as 0 for `n<2` rather than undefined) and a z-score outlier flag per value (`--outlier-stddev`, default 2.0 — requires `n>=3` and `stddev>0`, since flagging with fewer samples is meaningless). "Group" is workload command (`--group-by`'s default), `hostname`, or `cpu_vendor` — a fixed whitelist validated by `parse_group_by()` before being interpolated into the group-column position of the query (the only place a `--group-by` value reaches raw SQL). `--command`/`--hostname` filter which runs contribute (substring/exact match, bound as query parameters); `--metric` (repeatable) filters which metrics are reported, applied client-side after the query since SQLite parameter binding doesn't cover an `IN (...)` list cleanly. A bucket with fewer than `--min-runs` contributing runs is skipped (counted, not fatal — the same degrade-don't-fail idiom as `coverage.c`/`provenance.c`); `--strict` exits 1 if any bucket was skipped this way or if nothing matched at all. `open_summary_db()` refuses a database whose `PRAGMA user_version` predates `store.c`'s `metric_values` table (schema version 2, `MIGRATION_V1_TO_V2`) but otherwise tolerates a newer schema version, since store.c's own versioning convention never removes/renames a column on a MINOR bump. `--show-runs`/`--trace` are `INVESTIGATION_4.0.md`'s 4.1 Tier 2 item 14, "Traceability links (summary row → manifest → raw CSV → plots → tree artifacts)" — the other criterion deferred from 4.0 alongside the summary generator above (see that doc's "Success criteria for a 4.0 kickoff"). `--show-runs` appends every contributing run's `hostname:run_id` to a bucket (all of them, not just `outlier_run_ids`' flagged subset, via the same `append_list_entry()` truncate-not-overflow helper `outlier_ids` now also shares) as a new trailing column, giving a surprising summary number a concrete run identity to chase; a bucket with enough runs to overflow the column's buffer gets a `(+N more)` marker rather than silently losing the tail, since (unlike `outlier_ids`, normally just a handful of flagged runs) a `--show-runs` list is meant to be complete. `--trace <hostname>:<run_id>` is a standalone mode (ignores every grouping/filter option, only needs `--db`) that resolves that identity via `trace_run()`, a direct `runs` table lookup by `(hostname,run_id)`, and prints the run's command/environment fields plus `stat()`/`opendir()`-checked existence for `store.c`'s recorded `manifest_path`/`output_path`/`tree_output_path` columns (one small loop over a `{path_key,exists_key,column}` table, not three copy-pasted blocks) and a derived `<output_path's dir>/plots` directory (`wspy-plot`'s output location, not a column store.c tracks anywhere; a bare relative `output_path` with no `/` at all has no derivable directory, so this degrades to `exists=0` rather than guessing a `plots` path relative to wherever `wspy-summary` happens to be running from) — closing the actual "manifest → raw CSV → plots → tree artifacts" chain from a run identity, degrading each field independently (`exists=0`) rather than failing when a path no longer resolves (e.g. a run-index ingested from a different host, per `doc/ARTIFACT_CONTRACT.md`'s note that `manifest_path`/`output_path` are frequently host-relative in that setup). Output is stable `key=value` lines (embedded `\n`/`\r` in a value replaced with a space, since the format is one record per line), not `--csv`'s table shape or a bespoke JSON encoding, so a caller (`web/server.py`, a script) can parse a single resolved run without a JSON library on either side. Returns exit 1 (not 2) when the `(hostname,run_id)` pair isn't in the store at all, mirroring `--strict`'s "still needs more data, not a usage error" convention. `#ifndef TEST_SUMMARY` wraps only `usage()`/`main()`, matching `store.c`/`ledger.c`/`validate.c`, so `test_summary.c` can `#include "summary.c"` directly.
- `plot.c` — `wspy-plot` binary: the "Shared plotting templates" item from `INVESTIGATION_4.0.md`'s 4.1 Tier 2 list (item 12), replacing `workload/phoronix/gnuplot.sh` (retired) -- that script only knew two literal filenames (`amdtopdown.csv`/`systemtime.csv`) at fixed column positions; this generalizes to any wspy CSV output via the same normalized-schema convention `store.c`'s `ingest_csv_metrics()` established (a column named exactly `time`/`core`/`phase` is a dimension, every other non-empty-named column is a metric — column *identity*, not position or which flags produced it, decides what a CSV holds). A small built-in `templates[]` table (`--list-templates` to print it) pairs a counter group's known metric column names (e.g. topdown's `retire,frontend,backend,speculate`, straight from `topdown.c`'s own `PRINT_CSV_HEADER` text) with a title/ylabel/minimum-match count; `match_templates()` fires a template once enough of its candidate columns are present in a CSV's header (extra columns a firing template doesn't ask for, e.g. a newer wspy's `confidence,sanity` after topdown's core four, don't block the match), and `add_fallback_match()` sweeps up any remaining metric columns no template (or the network-io bucket below) claimed into one generic "metrics" plot, so an unfamiliar counter-group combination still produces something rather than nothing — except `counters_measured`/`counters_requested` (`coverage.c`'s per-row bookkeeping, present regardless of which counter groups were requested), excluded from every match via `is_excluded_metric_column()` since they're not a per-tick workload measurement. `add_network_fallback_match()` runs between the two: `system.c`'s per-interface byte counters (`net <iface>`, one column per interface discovered on this host, so no fixed template entry can name them) get their own "network-io" plot rather than falling into the generic fallback, since their byte-count magnitude otherwise dwarfs `--system`'s load-average/percentage columns on the same linear axis and flattens them to an indistinguishable line — confirmed empirically against a real `systemtime.csv`, not a hypothetical. Only a CSV with a `time` column (i.e. produced with `--interval`) is a time series to chart at all — an aggregate CSV is silently skipped, not an error. `wspy-plot --rundir <dir>` scans every `*.csv` directly in `<dir>` (`--csv <file>` also accepts one explicitly) and writes matched plots into `--out-dir` (default `<rundir>/plots`, `wspy-run`'s previously-empty reserved directory — see its own entry below) via a generated gnuplot script piped to `gnuplot`'s stdin (`set terminal png ... noenhanced` -- gnuplot's default "enhanced" text mode otherwise renders a literal `_`/`^` in an autotitled column name, e.g. `counters_measured`, as subscript/superscript markup instead of the literal character), one PNG per matching template per CSV, named `<csv-stem>.<template-name>.png`. A gnuplot invocation that's actually attempted and fails is the only real failure (exit 1); a CSV with no `time` column or no template match is not (exit 0) — the same "measured vs unavailable" degrade-don't-fail idiom used throughout this codebase.
  `--plot NAME=col1,col2,...` (repeatable) is the escape hatch for when the built-in templates' groupings don't fit what a user actually wants charted together — e.g. two counters from different built-in templates that happen to share a scale, or a workload-specific pair of columns nothing above knows about. `parse_custom_plot_spec()` parses the `NAME=col1,col2,...` argument text once at startup (a malformed spec, e.g. no `=` or no columns, is a usage error, exit 2 — caught up front rather than discovered per-CSV later); `add_custom_plot_match()` then matches each spec against every CSV scanned, including only the named columns that are actually present in *that* CSV's header (a named column absent from a given CSV is warned about and skipped, not fatal — the same graceful-degradation idiom as everywhere else here; a spec with none of its columns present produces no plot for that CSV at all). Unlike a built-in template, one present column is enough to fire — there's no "not enough of them" threshold, since the user asked for these specific columns. `--plot` is additive by default (its columns are also still eligible for a built-in template/fallback match, e.g. asking for `retire` in a custom plot doesn't stop `topdown` from also including it); `--only-custom` (requires at least one `--plot`) instead skips every built-in template and both fallback buckets, rendering exactly the given spec(s) and nothing else. `build_plot_matches()` is the pure, gnuplot-free half of this decision (custom specs, then — unless `--only-custom` — `match_templates()`/`add_network_fallback_match()`/`add_fallback_match()`), factored out of `process_csv()` specifically so `test_plot.c` can exercise every matching combination (including `--only-custom`) without needing gnuplot installed; `process_csv()` itself just calls it once per CSV and then loops rendering whatever it returned. `render_match()` removes `out_png` if `pclose()` reports a nonzero exit (e.g. every data point undefined because the requested counters had no perf access this run, or simply weren't collected) — gnuplot's own `set output` already creates the file before the `plot` command that can still fail, so without this a failed render left a stale 0-byte PNG sitting in `plots/` for a report to link to as if it had succeeded. `#ifndef TEST_PLOT` wraps only `usage()`/`main()`, matching `store.c`/`ledger.c`/`validate.c`/`summary.c`, so `test_plot.c` can `#include "plot.c"` directly and drive `match_templates()`/`add_network_fallback_match()`/`add_fallback_match()`/`add_custom_plot_match()`/`build_plot_matches()`/`process_csv()` against synthetic CSV header strings without gnuplot installed.
- `coverage.c`/`coverage.h` — counter capability discovery + "measured vs unavailable" coverage reporting (`INVESTIGATION_4.0.md`'s "Reproducibility, comparability, statistics" track — see "What shipped in 4.0"). `topdown.c:setup_counters()` calls `coverage_note()` for every counter it attempts to open via `perf_event_open`, success or failure, accumulating the global `coverage_requested`/`coverage_measured` counts and a `coverage_entries` list. A failed counter no longer aborts the run (`setup_counters()` used to call `fatal()` on any single `perf_event_open` failure — this made partial counter loss, e.g. from `perf_event_paranoid`/NMI-watchdog contention, indistinguishable from "wspy doesn't work here"); it's now just logged and reflected in coverage instead, so a run degrades gracefully rather than producing no output at all. `print_counter_coverage()` prints a concise "N/M measured" summary (plus a gap list in human output; `counters_measured,counters_requested` columns in CSV) alongside a real run's metrics, and the same counts/gaps are written into `--manifest`'s `counter_coverage` object and `--run-index`'s leaner `counter_coverage` counts-only field. `wspy --capabilities` is the standalone discovery half: it needs no workload command, forces `counter_mask` to `COUNTER_ALL` (ignoring any other counter flags — probing everything is the point), and prints `print_capability_report()` (every counter, available or not) before exiting. It also calls `ibs.c`'s `ibs_probe()`/`print_ibs_capability_report()` as part of the same report (see `ibs.c` below).
- `preflight.c`/`preflight.h` — counter-fit preflight: estimates, from the counter groups a `counter_mask` would request, whether they'll fit in the available general-purpose hardware PMU counter slots without multiplexing, *before* any `perf_event_open()` calls are made (`INVESTIGATION_4.0.md`'s "Existing-capability extensions" track, "Counter-fit preflight" item — builds on availability/NMI-watchdog handling and `coverage.c` that already exist at runtime, surfacing the same fit information before a run instead of discovering it after, e.g. as silent undercounting since `read_counters()` doesn't yet scale by `time_running`/`time_enabled`). `preflight_evaluate_groups()` walks an already-built `counter_group_list` and sums, per group, only the counters that land on the per-core general-purpose PMU — `PERF_TYPE_HW_CACHE` groups count in full, `PERF_TYPE_RAW` groups count only entries whose per-counter `device_type` is itself `PERF_TYPE_RAW` (excluding AMD's L3 escape-hatch entries and AMD IBS's dynamic PMU types, both of which have their own separate uncore/system budget) — against an available-slot estimate (`nmi_running ? 5 : 6`, the same heuristic `topdown.c`'s AMD raw/cache group chunking already bakes in, kept in sync rather than inventing a second number). `preflight_evaluate()` is the convenience wrapper for real use, temporarily swapping the global `counter_mask` and calling `setup_counter_groups()` for a throwaway list (same idiom `wspy.c`'s `run_capabilities_probe()` uses for `--capabilities`) — needs no privileges and opens no perf events, since it's pure arithmetic over the event tables. `wspy.c:main()` calls it automatically right after `setup_raw_events()`, before any counter is opened, and `preflight_warn_if_tight()` prints a stderr warning (silent when the fit is fine, mirroring `check_nmi_watchdog()`'s style) with suggested downgrades (largest contributor(s) first, via a `COUNTER_*` bit → `--no-<flag>` hint table) and, when the NMI watchdog is the reason a run doesn't fit, an explicit tip that stopping it (`scripts/setup_perf.sh` or `sudo sysctl -w kernel.nmi_watchdog=0`) frees 1 more slot. The same check is also available standalone via `wspy --preflight [<counter flags>]`, which evaluates the given flags (not `COUNTER_ALL` — unlike `--capabilities`, the point is to check the combination the caller intends to run) and prints the full per-group breakdown via `print_preflight_report()` without launching a workload.
- `multipass.c`/`multipass.h` — the pure, unit-testable half of native multi-pass counter execution (`wspy --passes=<list>`, `INVESTIGATION_4.0.md`'s 4.1 Tier 1 "Native multi-pass counter execution" item; the actual per-pass execution loop, `run_multipass()`, lives in `wspy.c` since it's coupled to that file's own execution-state globals — see `wspy.c`'s entry above). `multipass_group_names[]` maps each `--passes=<list>` token to a `COUNTER_*` bit, in ascending bit-value order (`wspy.h`'s own existing order) — that order is also the deterministic bin-packing iteration order, so `--passes=cache2,ipc` and `--passes=ipc,cache2` always produce the identical plan regardless of argv order. `multipass_plan_build()` greedily bin-packs the requested bits into passes that each fit the available general-purpose hardware PMU budget, reusing `preflight_evaluate()`'s real `setup_counter_groups()`-backed arithmetic at each step (not a hand-summed per-bit cost) — this correctly handles groups that share an underlying counter (e.g. `--dcache`/`--icache`/`--tlb` all reuse one `"instructions"` event in `cache_events[]`, `topdown.c`) rather than double-counting it. `COUNTER_SOFTWARE` is a deliberate special case: since `setup_counter_groups()` never builds a group for it itself (software counters are added separately, by both `main()`'s single-pass path and `run_multipass()`), it never registers any cost against the preflight budget — folded into whichever pass is already accumulating (or given a trivial pass of its own if none is open) rather than let a `preflight_evaluate()` call that can't see its true zero cost bounce it into a spurious extra re-execution of the workload. A single group that alone exceeds the budget still gets its own (multiplexing) pass, with the same warning `preflight_warn_if_tight()` already prints for a tight single-pass run, rather than blocking the whole feature. `COUNTER_IBS` is deliberately absent from `multipass_group_names[]` — IBS is fatal'd against `--passes` in `wspy.c` (its own separate system-wide PMU budget, already excluded from `preflight.c`'s general-purpose count, doesn't compose with per-pass bin-packing). `multipass_plan_build_multiplexed()` is the `--multiplex` counterpart to `multipass_plan_build()`: always exactly one pass covering the entire requested mask (no bin-packing), still routed through `preflight_evaluate()`/`preflight_warn_if_tight()` so a run that won't fit warns the same way a naturally-oversized single group already does in the bin-packed path.
- `phase.c`/`phase.h` — interval automatic phase-boundary detection: classifies each `--interval` tick as `warmup`/`steady`/`degraded` from that tick's IPC (`INVESTIGATION_4.0.md`'s "Existing-capability extensions" track, "Interval (`--interval`) → automatic phase-boundary detection" item — the basic-marker-detection half; named prerequisite for phase-aware topdown/IBS, 4.2). Signal is per-tick IPC (instructions/cpu-cycles, already scaled for multiplexing by `read_counters()` — see `topdown.c`'s entry above — by the time `phase_current_ipc()` reads it walking the "ipc" `counter_group` on `cpu_info->systemwide_counters` for `find_ci_label(cgroup,"cpu-cycles"|"instructions")`; it only checks `time_running != 0` to detect a tick where the counter wasn't scheduled at all, not to rescale). `phase_detector_update()` is a small, deliberately simple state machine (see `phase.h`'s header comment for the exact thresholds): warmup ends once a fixed-size rolling window's coefficient of variation drops below a threshold, recording the window mean as the steady-state baseline; a sample landing far enough below baseline is a *candidate* degraded transition that only commits after N consecutive candidate ticks (avoids one noisy sample flipping the label), with a hysteresis gap on recovery back to steady so a sample sitting right at the boundary doesn't flap the phase every tick; the baseline itself EMA-drifts while steady so slow legitimate drift isn't mistaken for degradation. `phase_detect_is_available()` gates the whole feature on `interval && phase_flag && (counter_mask & COUNTER_IPC) && !aflag` — `--per-core` (`aflag`) is excluded because its counters live on each core's own list, which `--interval`'s `timer_callback()` never reads at all (a separate, pre-existing `--per-core` + `--interval` CSV-shape gap noted in `tests/capability_matrix.sh`'s `per-core-topdown` bundle comment). The single run-lifetime `phase_state` (`struct phase_detector`) is initialized once in `wspy.c:main()` right before CSV headers are written, then fed once per tick from both `topdown.c`'s `timer_callback()` and `wspy.c`'s own final (tail-interval) read, so the last CSV row/human print reflects the same classification a periodic tick would have. Output is a `phase` CSV column immediately after `time` (present only when `phase_detect_is_available()`, so `--no-ipc`/`--per-core`/`--no-phase-detect` interval runs keep the pre-existing narrower header) plus, in human (non-CSV) output only, a `phase boundaries:` summary listing each transition with its elapsed-time timestamp (`phase_print_boundaries()`) — CSV output deliberately has no second, redundant representation of the boundaries, since a downstream reader can already reconstruct them by diffing adjacent rows' `phase` column. `--no-phase-detect` (default `phase_flag = 1`, paired `--phase-detect`) opts out. Not yet implemented: persisting phase boundaries into `--manifest`/`--run-index` — deferred until an actual consumer (phase-aware topdown, 4.2) needs them directly rather than via the CSV.
- `ibs.c`/`ibs.h` — capability-driven discovery of AMD's Instruction-Based Sampling (IBS) PMU support, plus the `ibs-basic`/`ibs-memory-deep` collection profiles built on top of it (`INVESTIGATION_4.0.md`'s "Zen5 / IBS" track). IBS is exposed as two dynamic PMUs the kernel registers at boot, `ibs_fetch`/`ibs_op`, present only when the CPU/kernel combination supports it; `ibs_probe()` reads `/sys/bus/event_source/devices/{ibs_fetch,ibs_op}/{type,format/*,caps/*}` at runtime (readdir-driven, not a hardcoded field list) rather than assuming which format fields (e.g. `l3missonly`, `ldlat`, `rand_en`) or caps (e.g. `zen4_ibs_extensions`, `dtlb_pgsize`) exist, since both vary by kernel version and CPU generation. Returns a `struct ibs_capabilities` (never fails; an absent PMU just yields `present`/`supported` 0) that `wspy --capabilities` prints via `print_ibs_capability_report()` alongside the existing counter capability report; `ibs_pmu_format()`/`ibs_pmu_cap()` are by-name lookup helpers into it.
  `--ibs-basic`/`--ibs-memory-deep` (`wspy.c`, setting `counter_mask |= COUNTER_IBS` and the module-global `ibs_collection_profile`) actually open IBS as counting events — `ibs_build_fetch_event()`/`ibs_build_op_event()` assemble a real `perf_event_attr` config by parsing each requested format field's sysfs `location` string (e.g. `"config1:0-11"`) into a (word, bit-range) and OR-ing the value in, rather than hardcoding bit offsets; `ibs_counter_group()` wires the result into an ordinary `struct counter_group` using the same `type_id == PERF_TYPE_RAW` + per-counter `device_type` escape hatch `raw_counter_group()` uses for L3 events (since `ibs_fetch`/`ibs_op` each have their own dynamic `type`, unlike a group's shared `type_id`). Both profiles read `ibs_fetch`/`ibs_op` unfiltered; `ibs-memory-deep` additionally requests `l3missonly`+`ldlat` filtering on `ibs_op` (and `l3missonly`+`fetchlat` on `ibs_fetch`) and opens a second, always-unfiltered `ibs_op_unfiltered` counter purely as a baseline. `struct ibs_event`'s `*_requested`/`*_applied` pairs distinguish "the profile calls for this filter" from "this kernel/CPU actually exposed the format field to encode it" — `topdown.c`'s `print_ibs()` surfaces both: the `ibs_op_unfiltered`/`ibs_op_accepted_ratio` CSV columns (and `ibs_l3missonly`/`ibs_ldlat_threshold`/`ibs_fetchlat_threshold`) are the sampling-skew/quality annotation the doc calls for — L3-miss-only/load-latency filtering is documented to skew the effective IBS sampling period, so that skew is visible in output instead of only in a code comment — and a requested-but-not-applied filter prints an explicit warning rather than silently collecting unfiltered data under a "deep" label. IBS collection is system-wide only (`wspy.c:main()`, alongside the software-counters block) — it's a machine-wide dynamic PMU, not a per-core countable event, so `--per-core` doesn't apply to it. `--ibs-maxcnt`/`--ibs-ldlat`/`--ibs-fetchlat` override the built-in default thresholds (`IBS_DEFAULT_*` in `ibs.h` — wspy's own starting points, not hardware-mandated values). `wspy-run`'s `ibs-basic`/`ibs-memory-deep` builtin profiles wrap these into single-pass profile-launcher invocations.
- `provenance.c`/`provenance.h` — best-effort environment/provenance capture (`INVESTIGATION_4.0.md`'s "Reproducibility, comparability, statistics" track: host/guest role, kernel, microcode, BIOS/power, governor, memory profile, tool versions). `provenance_collect()` is called once in `wspy.c:main()` right before `write_manifest`/`append_run_index`, populating a `struct provenance_info` with the CPU's virtualization role (`cpuid` leaf 1's hypervisor-present bit, plus the leaf `0x40000000` vendor string when a guest), microcode version (`/proc/cpuinfo`), BIOS vendor/version/date (`/sys/class/dmi/id/*`), `cpu0`'s frequency governor/scaling driver (plus a `cpu_governor_uniform` flag checked against every other online core), total memory (`sysinfo(2)`), and the toolchain that built this binary (`__VERSION__`, `gnu_get_libc_version()`). Each of the 9 `PROVENANCE_TRACKED_FIELD_COUNT` fields degrades to unavailable-with-reason independently rather than failing the run — mirrors `coverage.c`'s "measured vs unavailable" pattern, per the same doc's recommendation to treat environment fields as optional-but-recommended rather than blocking publishability. `manifest.c` emits the full field set plus a gap list under `environment`/`environment_coverage`; `run_index.c` emits the same fields in its leaner compact form plus counts-only coverage, mirroring how `counter_coverage` is handled in each. x86_64-only (`cpuid`), like the rest of this codebase's CPU-vendor detection (`cpu_info.c`).
- `wspy-run` — bash profile-driven launcher wrapping `wspy`, not part of the C build (no compilation, just `chmod +x`). Runs one or more named "passes" (each a separate `wspy` invocation with its own flag set) against a single workload command, either from a builtin profile (`quick`, `deep-cpu`, `deep-cpu-intel`, `deep-gpu`, `tree-heavy`, `ibs-basic`, `ibs-memory-deep` — a comma-separated list like `deep-cpu,tree-heavy` composes more than one profile's passes into a single invocation) or a `-c <file>` pass-list config file, writing each pass's output to `<outdir>/<prefix><pass-name>.<csv|txt>` (extension picked by whether that pass's flags include `--csv`). This is the "Common workload wrapper / profile-driven launcher" `INVESTIGATION_4.0.md` item — it collapses the repeated multi-invocation command lines historically hand-written in `workload/*/run_test.sh` (see below) into reusable profiles, and it's the lightweight precursor to the full config-driven experiment system (4.3), not that system itself. `--manifest-dir`/`--run-index` pass those `wspy` flags through per-pass, reusing `manifest.c`/`run_index.c` rather than inventing a second run-record format. `run_pass()` also passes `--config-name <pass-name>` on every invocation and, when `PROFILE` (a named builtin profile, not a `-c`/`--config` pass-list file) is set, `--preset-name "$PROFILE"` — structured configuration provenance (`INVESTIGATION_4.0.md` item 16), recorded into each pass's own manifest/run-index via `manifest.h`'s `config_provenance` field (see that entry above) rather than left implicit in the pass's flag string. Each pass is a separate `wspy` process — native single-process multi-pass execution (running several counter groups across automatically-sized internal passes within one launch of `wspy`) now exists as its own lower-level capability (`wspy --passes=<list>`, `multipass.c`/`multipass.h`, `INVESTIGATION_4.0.md`'s "Native multi-pass counter execution", 4.1), but `wspy-run`'s own builtin profiles (`deep-cpu` et al.) still shell out to N separate `wspy` invocations as before — collapsing those profiles' counter-sweep passes onto `--passes` is a documented follow-up, not part of that item. Note `wspy`'s own exit status reflects whether `wspy` itself ran, not the launched command, unless `--exit-with-child` is passed (opt-in, since existing scripts may rely on `wspy` always exiting 0 regardless of the child — it exits with the child's own exit code, or 128+signal if the child was killed by a signal, using `child_exit_known`/`child_exit_code`/`child_signaled`/`child_term_signal` declared in `wspy.h` and populated by `wspy.c`'s `wait4()` path or `topdown.c`'s `ptrace_loop()` for `--tree`). `wspy-run` passes do not set this flag today, so a pass's own exit code still can't be used to detect workload-command failure — only `--manifest`/`--run-index` `exit_status` can, unless the flag is added to a pass's flag string by hand. Each builtin profile also carries a per-pass `PASS_TIMEOUTS` entry (empty by default, meaning unbounded) that `run_pass()` wraps the `wspy` invocation in via the `timeout` command when set — today only `tree-heavy`'s one pass sets one (3600s). That cap exists because an hour's worth of `--tree` process records is already more data than is practical to publish/browse, **not** because ptrace tracing itself runs slower than the counter passes; the counter passes are deliberately left unbounded, since a workload that legitimately takes a while to finish is fine to just wait out. `--list` shows `[timeout Ns]` next to any pass that carries one. This is a per-*pass* cap wrapped around one `wspy` invocation, not a cap on the whole `wspy-run` invocation — a composite profile like `deep-cpu,tree-heavy` can have some passes bounded and others not.
  Passing `--suite <name>` and `--benchmark <name>` together switches from that flat `<outdir>/<prefix><pass>.<ext>` naming to the **unified output layout** (`INVESTIGATION_4.0.md`'s "Run artifact foundation" track, "Unified output layout" item): every pass writes into one directory, `<outdir>/<suite>/<benchmark>/<run-id>/`, where `<run-id>` is an auto-generated sortable `<timestamp-ms>-<pid>` (same shape as `run_index.c`'s per-process `run_id`, but a distinct value — it identifies the whole `wspy-run` invocation, not one `wspy` process) unless overridden with `--run-id`. `--prefix` is rejected together with `--suite`/`--benchmark` (the directory already namespaces output), and `--manifest-dir` defaults to that same run directory so every pass's `<pass-name>.manifest.json` lands next to its output. Inside the run directory, `wspy-run` adds three things beyond the raw per-pass files: `summary.txt` (concatenation of every non-CSV, non-`--tree` pass's output, generalizing the `cat *.txt > combined.txt` step `workload/cpu2017`/`workload/phoronix` used to hand-roll), `manifest.json` (a small run-level index — `suite`/`benchmark`/`run_id`/`command` plus each pass's name, output filename, per-pass manifest filename, and wspy-level `status`; not the same document as a per-pass `--manifest`, which already exists per pass), and an empty `plots/` directory — `wspy-run` itself never writes into it, but it's the exact directory `wspy-plot --rundir <rundir>` (item 12, `plot.c`) populates when a caller runs it afterward, as `workload/phoronix/run_test.sh`/`workload/cpu2017/run_test.sh` and `web/server.py` all now do. A `--tree` pass (e.g. `tree-heavy`) writes `process.tree.txt` at the run directory root because its flags reference `${OUTDIR}` at profile-load time, which is already the run directory by then.
- `wspy-queue` — Python job/queue processor, alongside `wspy-run` (stdlib-only, `chmod +x`, no build step — same "thin client" principle as `web/server.py` below). This is `INVESTIGATION_4.0.md` item 13 ("Deployment/hosting design note"), which landed as a real feature rather than staying design-only: a **job** is a portable, spec-only JSON document (`web/joblib.py`'s `build_job()`/`validate_job()`) capturing "what should run" — workload command, suite/benchmark identity, and either a `wspy-run` preset spec or the same checklist object `web/server.py`'s Run tab posts — captured *before* any run directory or output exists and carrying no reference to the machine that created it (no absolute `--output-root`, no path into that machine's `--run-index`/`store.db`), so a job file can be copied verbatim to a second machine with wspy checked out and processed there against that machine's own independent output tree. A job moves through a `pending` → `running` → `done`/`failed` lifecycle implemented as literal `<jobs-dir>/<state>/<job-id>.json` directories (Maildir-style); claiming a job for processing is one `os.rename()` from `pending/` to `running/`, atomic on a POSIX filesystem, which is also the only concurrency control needed since jobs are processed strictly serially (no `--parallel` — a `wspy` run has exclusive use of the machine's hardware PMU counters, matching `preflight.c`/`coverage.c`'s existing single-process-concern treatment of counter contention). A failed job stays in `failed/` until `wspy-queue requeue <job-id>` moves it back — no silent auto-retry, matching the degrade-don't-fail-but-don't-hide-it convention `coverage.c`/`validate.c` use elsewhere. Subcommands: `add` (creates a pending job — preset mode via `--profile`, custom mode via `--checklist-json <file>`, both also taking `--custom-plot`/`--only-custom`/toggle flags mirroring `wspy-plot`'s and the web UI's own vocabulary), `run` (processes pending jobs — `--job <id>` for one specific job, `--limit N`, `--strict` to exit 1 on any failure; drains everything currently pending by default, so a plain cron invocation works as a queue worker with no daemon/watch mode needed), `list`, `show <job-id>`, `requeue <job-id>`. All the actual command construction and execution (checklist/preset → `wspy` argv, invoking `wspy`/`wspy-run`/`wspy-plot`/`wspy-store`, writing a `wspy-run`-shaped `manifest.json`/`summary.txt` for a custom-mode run) is shared with `web/server.py` via `web/joblib.py` (see that module's own docstring) rather than reimplemented — `wspy-queue` is a second front end over the same execution logic, headless and with no dependency on `web/server.py` being up. `web/server.py`'s Run tab is the primary job *creator* (a "Queue this instead of running it now" checkbox routes the same preset/checklist/workload state through `POST /api/enqueue-job` instead of launching immediately), while `wspy-queue` is the job *runner*; either tool can also do the other's job's own subset (`wspy-queue add` creates jobs too, for a fully browser-free workflow). Tested by `tests/wspy_queue_smoke.sh` (fake `wspy`/`wspy-run`/`wspy-plot`/`wspy-store` binaries — lifecycle, failure/requeue, and portability via copying a job file into a second, independent jobs/output tree) and `web/test_joblib.py` (job schema/validation and checklist→argv logic, standalone `python3 web/test_joblib.py` — not wired into `make test`/`./run_tests.sh`, matching `web/`'s own "not covered" convention below).
- `web/` — `server.py`: `INVESTIGATION_4.0.md`'s 4.1 Tier 2 items 6-9 — a tabbed Run/Validate/Store & Summary/Discovery launcher plus a report browser that curates. Stdlib-only Python (no dependency, no build step — matches the "thin client over existing commands" design principle the whole tier is held to), a `ThreadingHTTPServer` bound to `127.0.0.1` by default. `web/joblib.py` (item 13's addition) holds the checklist/preset → `wspy` argv builders (`build_configuration_passes()`/`build_wspy_run_argv()`/`build_pass_argv()`/etc.) and the actual run executors (`execute_profile_run()`/`execute_custom_run()`) that used to live directly in `server.py` — pulled out, with no behavior change, so `wspy-queue` (see its own entry above) can share the identical logic instead of reimplementing it; `server.py` imports everything it needs back from `joblib.py` and is otherwise unchanged. See `joblib.py`'s own module docstring for the full breakdown, including the job-file format (`build_job()`/`validate_job()`) item 13 added. `GET /` renders four tabs (`render_run_tab()`/`render_validate_tab()`/`render_store_tab()`/`render_discovery_tab()`, wired via `.tab-btn`/`.tab-panel` in `web/static/app.js`); items 6/7's two separate homepage forms (below) were folded into item 9's single Run tab once it existed, though their backend endpoints/report renderers stay so old reports and "customize & run again" links keep working.

**Key data structures** (`cpu_info.h`):
- `struct cpu_info` — root descriptor: vendor/family/model, core count, `systemwide_counters` or per-core counters
- `struct counter_group` — linked list of related counters (IPC, topdown, branch, cache, ...), chained via `.next`
- `struct counter_info` — one perf counter: fd, label, config, running/enabled times, `is_group_leader` flag
- `struct raw_event` — vendor-specific descriptor (`intel_raw_events[]`/`amd_raw_events[]` tables in `topdown.c`), parsed by `parse_intel_event()`/`parse_amd_event()` at startup into a `.raw.config` value via the `union intel_raw_cpu_format`/`union amd_raw_cpu_format` bitfields

**Counter mask bits** (`wspy.h`): `COUNTER_IPC`, `COUNTER_TOPDOWN[2]`, `COUNTER_TOPDOWN_{FE,BE,OP}`, `COUNTER_BRANCH`, `COUNTER_{D,I,L1,L2,L3}CACHE`, `COUNTER_MEMORY`, `COUNTER_TLB`, `COUNTER_OPCACHE`, `COUNTER_SOFTWARE`, `COUNTER_FLOAT` — combined into the global `counter_mask`, set via CLI flags in `parse_options()`. `system_mask` (`wspy.h`: `SYSTEM_LOADAVG|SYSTEM_CPU|SYSTEM_NETWORK` by default, `system.c`) is always populated; `--system`/`-s` (`sflag`) just controls whether `print_system()` is called, while `--gpu-busy`/`--gpu-metrics`/`--gpu-smi` additionally OR in `SYSTEM_GPU` regardless of `sflag`.

**Child launch protocol** (`topdown.c`): parent creates a pipe and forks; if `--tree` the child calls
`PTRACE_TRACEME` and blocks on the pipe; the parent sets up and starts counters, sleeps 2s, then writes
`"start\n"` to the pipe; the child reads it and `execve`s the real workload. With `--tree`, the parent
runs `ptrace_loop()` (setting `PTRACE_O_TRACEFORK|TRACEEXIT`) to record fork/exit events with timestamps
to the tree file; otherwise it just `wait4()`s on the child.

**GPU support (`AMDGPU=1` only):** two independent GPU paths, both gated by `system_mask & SYSTEM_GPU`
and only invoked when explicitly requested via CLI flag:
- `--gpu-smi` → `amd_smi.c` (ROCm `libamd_smi` library; includes `<amd_smi/amdsmi.h>`, so `ROCM_DIR/include/amd_smi/amdsmi.h` and `-lamd_smi` must exist under `ROCM_DIR`)
- `--gpu-busy` / `--gpu-metrics` → `amd_sysfs.c` (direct sysfs reads: busy %, temp, activity, power, freq)
When built without `AMDGPU=1`, these flags are still recognized by the option parser but print a
"GPU support not built" warning instead of erroring, so the binary behaves consistently either way.

**CSV vs. human output:** every `print_*` function switches on `enum output_format` (`PRINT_NORMAL`,
`PRINT_CSV`, `PRINT_CSV_HEADER`). When adding a column, the header case and value case must be added
together and stay in the same order — `run_tests.sh` checks exact CSV column ordering (e.g.
`elapsed,utime,stime,gpu_busy,ipc`), so verify column position when adding new metrics. Two related
pitfalls `tests/golden_output.sh`'s generic column-count check (and manual review) should catch: the
`PRINT_CSV` value row must emit exactly the columns declared in `PRINT_CSV_HEADER` with **every field
comma-terminated** (a bare `fprintf(outfile,"%4.1f",x)` with no trailing comma silently fuses into the
next field instead of erroring), and it must do so **unconditionally** — gating the value row behind
`if (some_counter_value)` (rather than only gating on vendor/group applicability) means a
permission-denied run silently drops every column in that row while the header still claims them. Both
classes of bug shipped in this codebase before being caught by writing `tests/golden_output.sh` itself
(see that commit's history for the concrete cases: `--software --csv`, `--topdown-frontend`,
`--topdown-optlb`).

## Common edits

**New raw perf counter:** add an entry to `intel_raw_events[]`/`amd_raw_events[]` in `topdown.c` with an
`event=...,umask=...` description string and the appropriate `COUNTER_*` mask from `wspy.h` (add a new
bit there if needed); the parser converts the description to `.raw.config` in `setup_raw_events()`. Add
a CLI flag pair (`--foo`/`--no-foo`) in `wspy.c:parse_options()` if it should be user-toggleable.

**New counter group:** write a constructor in `topdown.c` following the existing pattern (e.g.
`software_counter_group()`, `cache_counter_group()`), allocate `struct counter_group` + `struct
counter_info[]`, and link it into the list in `setup_counter_groups()` based on `counter_mask`. Each
check there *prepends* to `*counter_group_list` (`cgroup->next = *counter_group_list; *counter_group_list
= cgroup;`), so groups print in the reverse of the order they're checked in `setup_counter_groups()` —
place a new `if (counter_mask & COUNTER_FOO)` block earlier/later in that function to control where it
lands in output, and check nearby groups' relative order if you need it to appear next to a specific one.

**New system metric:** parse it in `system.c:read_system()`, store it in `struct system_state`, print it
in `system.c:print_system()` with matching CSV/header/normal cases, and add a `SYSTEM_*` bit + CLI flag
if it should be independently toggleable.

**New manifest field:** add it to `struct manifest_info` in `manifest.h`, populate it at the call site in
`wspy.c:main()` (near the end, guarded by `if (manifest_path || run_index_path)`), and emit it in
`manifest.c:write_manifest()`. Adding a field is a backward-compatible change — bump the MINOR component
of `MANIFEST_SCHEMA_VERSION`; removing or renaming one is a MAJOR bump, since existing readers may depend
on the old shape.

**New run-index field:** most fields come from the same `struct manifest_info` used by the manifest (see
above), so if the field already exists there just emit it in `run_index.c:append_run_index()` too. A
field that's specific to the index record (not the manifest) needs its own plumbing through
`append_run_index()`'s signature. Either way, bump `RUN_INDEX_SCHEMA_VERSION` in `run_index.h` (MINOR for
an added field, MAJOR for removed/renamed) — it's versioned independently of `MANIFEST_SCHEMA_VERSION`.

**New normalized-store field (`store.c`):** add the column to the relevant `CREATE TABLE` in
`SCHEMA_DDL` (the fresh-database shape, already at the current version), bind it in
`upsert_run()`/`upsert_run_environment()`/`enrich_from_manifest()`/`ingest_csv_metrics()` as
appropriate (most run/environment fields come straight from the run-index record already parsed
via `json_reader.h`; manifest-only host fields go through `enrich_from_manifest()`; per-row CSV
values go through `ingest_csv_metrics()`), and bump `STORE_SCHEMA_VERSION` — checked via `PRAGMA
user_version` on open, not a `store_meta` row. Also add a matching `ALTER TABLE`/`CREATE TABLE IF
NOT EXISTS` block to a new `MIGRATION_V<N-1>_TO_V<N>` DDL string and a new branch in
`ensure_schema()`'s version dispatch (`user_version == 1` → `MIGRATION_V1_TO_V2` is the existing
example to follow) — `ensure_schema()` runs *either* the fresh-DB `SCHEMA_DDL` (`user_version ==
0`) *or* exactly one migration step per existing older version, never both, so `ALTER TABLE ADD
COLUMN` never re-runs against a column that's already there. An old `wspy-store` binary still
refuses to write against a database whose `user_version` is newer than it understands, same as
before.

**New `wspy-run` builtin profile:** add a `case` arm in `load_builtin_profile()` in `wspy-run` with a
`PASS_NAMES`/`PASS_FLAGS` array pair (one entry per pass, flags as a single space-separated string), and
add it to the `BUILTIN_PROFILES` list at the top of the file so `--list`/`--help` pick it up. Keep pass
names filesystem-safe (they become part of the output filename) and don't rely on flag ordering within a
pass beyond what `wspy`'s `getopt_long` already tolerates.

**New `wspy-validate` sanity bound:** add an entry to `sanity_bounds[]` in `validate.c` (column name plus
a `{min,max}` pair) when a specific CSV column deserves a tighter check than the generic "finite,
non-negative, not implausibly large" rule every numeric column already gets. Percent-suffixed cells
(e.g. `"26.61%"`) are matched by their trailing `%`, not by column name, and use the separate
`PERCENT_SANITY_MAX` bound instead of this table.

**New AMD IBS filter (`ibs.c`):** don't hardcode a bit offset — add the lookup/apply call in
`ibs_build_fetch_event()`/`ibs_build_op_event()` using `ibs_pmu_format(pmu,"<field name>")`, matching the
existing `l3missonly`/`ldlat`/`fetchlat` pattern: set a `*_requested` flag unconditionally, then only set
the matching `*_applied` flag and call `apply_format_field()` if the field lookup succeeds, so a
kernel/CPU that doesn't expose the field degrades to "unfiltered" instead of failing. If the filter
should be visible as a skew/quality annotation (most are, since any op/fetch filtering changes what
counts as a "sample"), surface it in `topdown.c`'s `print_ibs()` CSV/normal output too, not just as an
internal field on `struct ibs_event`.

## Notable runtime behavior

- NMI watchdog: if `/proc/sys/kernel/nmi_watchdog` is active, one hardware counter is reserved system-wide; `check_nmi_watchdog()` in `topdown.c` detects this and reduces available counter slots accordingly rather than failing outright.
- AMD L3 cache events depend on `/sys/devices/amd_l3/type` and are silently skipped if absent.
- Vendor detection distinguishes AMD Zen/Zen5, Intel Atom/Core (hybrid), and ARM systems; on ARM, `cpu_info.c` also tracks PMU cluster/type per core so `--per-core` can bind raw counters to the correct PMU type on mixed clusters.
- `amd_sysfs_initialize()` scans `/sys/class/drm/card*/device/vendor` for AMD-vendor (`0x1002`) cards via `amd_sysfs_scan_devices()`, rather than assuming a fixed `card1` path — on a multi-GPU machine (e.g. an NVIDIA card enumerated as `card1` ahead of the AMD card at `card2`) the old hardcoded path silently read nothing; the scan fixes that. If no AMD card is found, `--gpu-busy`/`--gpu-metrics` warn and read zero rather than erroring. `--gpu-device=<idx>` (recognized regardless of `AMDGPU` build, like the other `--gpu-*` flags) selects a specific card/device by index for both the sysfs (`--gpu-busy`/`--gpu-metrics`) and SMI (`--gpu-smi`) backends — omitted or negative, the prior lowest-numbered-card/device-0 default applies; `wspy --capabilities` prints both backends' enumerated device lists (index + selection) on `AMDGPU=1` builds so the indices to pass are discoverable up front.
- `amd_sysfs_gpu_metrics()` branches on the `gpu_metrics` binary blob's `format_revision` (1/2/3, structs in `amd_sysfs.h`) since the layout is GPU-generation-dependent; unrecognized revisions are logged and ignored (`gpu_metrics_state.valid` stays 0, so callers must check `amd_sysfs_gpu_metrics_valid()` before reading temp/activity/power/freq). See `doc/sysfs.gemini.txt` for background on the blob format and where the kernel defines it (`kgd_pp_interface.h`).
